# requirement — projekt-lokale Verfahrens-Lessons (newest-first)

- **2026-07-05 · docs/-Writes im Worktree verifizieren (Split-Brain-Gefahr).** Beim Lauf in einem Git-Worktree (`.claude/worktrees/<x>/`) kann ein `Write` mit Worktree-absolutem `docs/`-Pfad trotzdem im **Haupt-Repo** landen, während die `board/`-Änderungen korrekt im Worktree liegen → stiller Split-Brain (Spec im Haupt-Baum, Stories im Worktree; eine Parallel-Session könnte die Spec im Haupt-Repo committen/verlieren). **Regel:** nach jedem Spec-/`docs/`-Schreibzugriff mit `git -C <worktree> status --short` prüfen, dass die Datei **im Worktree** als `??`/`M` erscheint; taucht sie stattdessen im Haupt-Repo (`git -C <main> status`) auf, in den Worktree verschieben und den Haupt-Baum sauber hinterlassen, **bevor** `board ready`/`lint` läuft (sonst falscher `SPEC-MISSING`-Fehler).
- **Entdeckte, nicht angeforderte Verbesserungspotenziale nie eigenmächtig als Item+Spec umsetzen** — ausschließlich im Output/Handoff eskalieren (Kern-Regel).
