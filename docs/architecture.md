# Detailkonzept / Architektur — dev-gui

> **Schicht 2 von 3.** Das **WIE konzeptionell** — logisch, sprach-/paradigma-unabhängig (Komponenten/Flows/Zustände, keine Idiome/Klassen). Bindend für den `coder`; Architektur-Konformität ist Review-Kriterium.

## Domänenmodell
- **Session** — die *eine* interaktive Claude-Code-Session, die die GUI fernsteuert. Lebenszyklus: `starting → ready → busy → ready` (bzw. `stopped`/`failed`). Genau eine pro Dienst.
- **Command** — ein ausgelöster Slash-Befehl (z.B. `/flow #12`) + Status (`queued → running → done|cancelled`) + Auslöser-Identität + Zeit.
- **Read-Models** (nur gelesen, nie persistiert): **Project** (Org-Repo ≠ agent-flow), **BoardItem**, **CIRun**, **PreviewContainer**.
- **AuditEntry** — append-only: Zeit, Access-Identität, Befehl.

## Komponenten
**Backend (Node, ESM, Express + ws):**
- **PtyManager** — startet/hält/restartet **genau eine** `claude`-Session in einem PTY (`node-pty`); schreibt Input, broadcastet Output; kennt den Session-Zustand. *Boundary:* einziger Ort, der den PTY berührt.
- **WS-Gateway** — WebSocket `/ws/terminal`: Client-Eingaben → PtyManager; PTY-Output → alle Clients. Reicht auch Status-Pushes durch.
- **CommandService** — nimmt Trigger entgegen, prüft **Allowlist** + **Concurrency-Lock (1)**, injiziert den Befehl in die Session, schreibt den **AuditEntry**. *Boundary:* einziger Schreibpfad in die Session von außen.
- **GitHubReader** — liest Projekte/Board/CI **und die Org-Repo-Übersicht** (Repo-Liste inkl. Sichtbarkeit + `htmlUrl`, [[github-repos-overview]]) über den GitHub-App-Token. *Boundary:* einziger **read-only** GitHub-Zugriff (kein POST/PATCH/PUT/DELETE). Liefert die Liste über `GET /api/github/repos` (alle Org-Repos, keine `agent-flow`/`dev-gui`-Ausschlüsse wie beim Fabrik-Dashboard).
- **GitHubWriter** *(neu — `src/GitHubWriter.js`; einziger **mutierender** GitHub-Boundary, getrennt vom read-only `GitHubReader`)* — führt schreibende GitHub-Aktionen aus: Org-Repository anlegen (`POST /orgs/{org}/repos`, [[github-repo-create]]) und bestehendes Repo lokal in `WORKSPACE_DIR` klonen ([[github-repo-clone]]). Mintet den Installation-Token **transient unmittelbar vor** dem Aufruf (App-ID/Installation-ID/Private-Key store-intern aus dem `CredentialStore`-Schema `github`); Token nie in Response/Log/Audit/WS/URL/Argv/persistierter Remote-URL. Mutierende Aktionen auditiert (Audit-First) + identitäts-/rollengeschützt (gleiche `CRED_ADMIN_EMAILS`-Logik wie ADR-007). **Betriebs-Vorbedingung Repo-Create:** GitHub-App braucht **Administration: Read & Write**. **Klon-Ziel** strikt innerhalb `WORKSPACE_DIR` (Path-Traversal-/Symlink-Schutz), Re-Clone nicht stillschweigend überschreibend. Detail-Architektur (SSH-/Git-Ausführungsweg, Token-Injektion ohne Persistenz) = `architekt`.
- **WorkspaceManager** *(neu — Boundary für `WORKSPACE_DIR`-Dateisystem + Git auf lokalen Klonen, [[workspace-repos]])* — scannt `WORKSPACE_DIR` **live** (direkte Unterordner mit `.git` = lokale Klone) und liefert pro Klon Branch/clean-dirty/letzter Commit/credential-freie `origin`-URL (`GET /api/workspace/repos`, read-only). Führt die mutierenden Workspace-Aktionen aus: **Pull** (`POST /api/workspace/repos/pull` — mintet **unmittelbar vorher** transient einen Installation-Token wie der Klon-Pfad; Token nie in Argv/Log/Audit/Response/persistierter `origin`-URL) und **Löschen** (`POST /api/workspace/repos/delete`). Alle Pfad-Operationen **strikt innerhalb** `WORKSPACE_DIR` (Path-Traversal-/Symlink-Schutz wie der Klon-Pfad). Mutationen auditiert (Audit-First) + identitäts-/rollengeschützt (gleiche `CRED_ADMIN_EMAILS`-Logik wie ADR-007). Ob dies eine eigene Komponente oder eine Erweiterung des Git-Ausführungs-Boundary aus [[github-repo-clone]] wird, entscheidet der `architekt`; bindend: **ein** Boundary für Workspace-FS/Git, Token nie persistiert. **Origin-URLs verlassen das Backend nur credential-frei.**
- **DockerReader** — liest Preview-Container über die Docker-Engine. *Boundary:* einziger Docker-Zugriff.
- **AccessGuard** — Middleware: validiert den Cloudflare-Access-JWT (`Cf-Access-Jwt-Assertion`) vor jeder `/api/*`- und WS-Anfrage; Fail-Fast beim Boot ohne Access-Konfig.
- **Static server** — liefert das gebaute React-Frontend.

**Frontend (React):**
- **App-Shell** — Einstiegs-Panel (vier Kacheln: GitHub · VPS · Cloudflare · Fabrik) + client-seitige Navigation (deep-linkbare Routen, Browser-Verlauf, Fallback auf Panel) + ein **Zahnrad** in der Navigation, das die **Settings-Ansicht** öffnet (Route `settings`, NICHT als fünfte Kachel). Rendert je nach Route eine der Ansichten. *Boundary:* einziger Ort, der View-Routing kennt.
- **Settings-Ansicht** — zentrale Einstellmaske mit Sektionen je Integration (GitHub · Cloudflare · VPS-Provider · SSH-Keys); Credential-Felder write-only/maskiert (nur Status „gesetzt/nicht gesetzt"). Konsumiert die Settings-/Credential-/SSH-Endpunkte (siehe unten).
- **Fabrik-Ansicht** — Terminal-Pane (xterm.js) · Status-Dashboard · Flow-Trigger-Panels · Job-/Kill-Steuerung (= bisheriges Frontend, jetzt als eine Ansicht eingebettet).
- **Cloudflare-Ansicht** *(v2 — Inventar + Lösch-Werkzeug, Capability A; ADR-010/011)* — listet die verwalteten Zones/Domänen, lässt eine Zone anwählen → zeigt deren Tunnel + Public-Hostname-Routen → erlaubt das **Löschen** einzelner Tunnel/Routen. Protected-Ressourcen (eigene `devgui`/Access-Mauer, ADR-011) werden als gesperrt markiert (kein Lösch-Affordance); jede andere Löschung verlangt einen **type-to-confirm**-Schritt (Hostname tippen). Konsumiert ausschließlich `/api/cloudflare/*`; **kein** Cloudflare-Token im Frontend.
- **Deployments-Ansicht** *(neu — Capability B, ADR-012; eigene Route `deployments`, nicht in die Cloudflare-View gedrängt)* — listet Deploys (Container+Route als Einheit) je VPS, bietet Deploy (Image+VPS+Hostname → atomare Einheit) und Undeploy; zeigt das Reconciliation-Ergebnis (Capability C) read-only an. Konsumiert `/api/deployments/*`.
- **GitHub- / VPS-Ansicht** — siehe oben (VPS = v2, ADR-009; GitHub = Platzhalter+Repo-Übersicht). *Geplant offen:* erweiterter GitHub-Schreibpfad.

**Backend — Cloudflare (Boundary in ADR-010/011/012/013 fixiert):**
- **CloudflareApi** *(neu — `src/cloudflare/CloudflareApi.js`; einziger Ort, der die Cloudflare-API via eingebautes `fetch` anspricht — analog `GitHubReader`/`VpsProviderRegistry`)* — kapselt Zones/DNS/`cfd_tunnel`-Endpunkte hinter einem schmalen, SDK-freien REST-Vertrag. **Read-Pfad** (Zones auflisten, Tunnel + Public-Hostname-Routen je Zone lesen) und **mutierender Pfad** (Tunnel-Konfiguration via `PUT …/cfd_tunnel/{id}/configurations`, DNS-Records anlegen/löschen, Tunnel/Route löschen) sind im selben Boundary, aber als getrennte Methoden-Gruppen (`read*` vs. `mutate*`). API-Token + Account-/Zone-Id store-intern aus dem `CredentialStore` (`credentials/cloudflare/api_token`, `…/account_id`; Zone-Id wird **live aus der Zones-API aufgelöst**, nicht als Credential gepflegt), transient pro Aufruf, **nie** in Response/Log/Audit/WS/Argv/Frontend-Bundle. Read-Aggregation **live** (kein Tunnel-Store, ADR-005-Linie), pro Zone degradierend. **Detail: ADR-010.**
- **LockoutGuard** *(neu — `src/cloudflare/LockoutGuard.js`; Self-Lockout-Hard-Block)* — reine, store-/netzlose Prädikat-Komponente, die für eine Route/DNS-Record/Tunnel entscheidet, ob sie **protected** (eigene `devgui`-Erreichbarkeit ODER Cloudflare-Access-Mauer) ist. Jeder mutierende Cloudflare-Pfad fragt **vor** der Aktion `LockoutGuard.isProtected(target)`; ist `true` → **422 `protected-resource`**, keine Mutation, egal welche Identität/welcher Confirm-Token. **Detail: ADR-011.**
- **DeployOrchestrator** *(neu — `src/deploy/DeployOrchestrator.js`; Boundary für die atomare Container+Route-Einheit, Capability B)* — orchestriert „ghcr-Image → Docker-Container auf einem VPS + Cloudflare-Tunnel-Route" als **eine** Einheit (Deploy) und das inverse Entfernen (Undeploy). Greift für den Container-Teil über die neue **VpsDockerControl**-Boundary auf den VPS zu, für den Route-Teil über `CloudflareApi`; bindet Container↔Route über eine **Label-Konvention** am Container (`cloudflare.tunnel-hostname`). Kein eigener Deploy-State-Store — Quelle der Wahrheit ist der Container (Docker-Label) ⊕ die Cloudflare-Tunnel-Config (live abgleichbar). **Detail: ADR-012.**
- **VpsDockerControl** *(neu — `src/deploy/VpsDockerControl.js`; einziger Ort für **schreibendes** Docker-on-VPS via SSH)* — `docker pull/run/rm` auf einem laufenden VPS über die `VpsProvisioner`-SSH-Linie (ADR-008). Read-only Docker bleibt beim lokalen `DockerReader`; diese Boundary deckt **remote, mutierend** ab (Deploy/Undeploy/Reconcile). **Detail: ADR-012.**
- **ReconciliationJob** *(neu — `src/deploy/ReconciliationJob.js`; node-interner Mitternachts-Scheduler, Capability C)* — gleicht periodisch je VPS laufende Container (`cloudflare.tunnel-hostname`-Label) gegen die Cloudflare-Tunnel-Routen ab, **beidseitig selbst-heilend** (Container-Label = autoritativer Desired-State): verwaiste Routen (kein managed Container) werden über `CloudflareApi` entfernt (nie protected); **managed Container ohne Route** bekommen die Route **angelegt** — über **denselben** atomaren Anlege-Pfad wie der `DeployOrchestrator` (ADR-012, nicht dupliziert). Erzeugt je Aktion eine im Cloudflare-Panel sichtbare interne Meldung (`ReconcileNotice`) und je Lauf (Cron + manuell) einen `ReconcileReport`, beide persistiert über die bestehende `AuditStore`-Mechanik (kein neuer Store). Läuft im bestehenden always-on dev-gui-Prozess (ADR-002), idempotent, Audit-First, pro VPS/Provider degradierend. **Detail: ADR-013.**

**Backend — geplant (Settings/Credentials; Boundary + Store ENTSCHIEDEN, s. ADR-007/008):**
- **CredentialStore** *(neu — `src/CredentialStore.js`; einziger Lese-/Schreibpfad zu `secrets.enc.json`)* — einziger Boundary für Geheimnisse (Credentials je Integration + SSH-Private-Keys). At rest **AES-256-GCM** auf dem persistenten Volume; Master-Key aus Env (`CRED_MASTER_KEY`, scrypt-abgeleitet). **Write-only nach außen:** Lese-Antworten liefern nur Metadaten (Status „set/unset", Maske, `updatedAt`), **nie** Klartext; Klartext verlässt den Store nur store-intern an Konsumenten. Endpunkte `GET/PUT/DELETE /api/settings/credentials*`, `GET/PUT/DELETE /api/settings/ssh-keys*`; mutierende Aktionen auditiert + identitäts-/rollengeschützt (Access-Identität genügt; optionale `CRED_ADMIN_EMAILS`-Allowlist). **Details + Datei-Schema: ADR-007.**
- **VpsProvisioner** *(neu, Folge-Capability — `src/VpsProvisioner.js`; Boundary in ADR-008 fixiert)* — einziger Ort für SSH-Verbindungen; trägt einen hinterlegten Public-Key idempotent in `authorized_keys` eines VPS-Ziels ein (`POST /api/settings/ssh-keys/{user}/provision`), Private-Key store-intern aus dem `CredentialStore`. Ziel-Schema `{ host, port?, targetUser }`. Detail-Spec = #47; #45/#46 dürfen keinen SSH-/Provider-Code einführen (s. ADR-008).
- **VpsProviderRegistry / VpsProvider-Adapter** *(neu, Multi-Provider — Boundary für Cloud-Provider-APIs; **Boundary in ADR-009 fixiert**)* — einziger Ort, der Provider-APIs (Hetzner, IONOS, Hostinger) anspricht; provider-agnostischer `VpsProvider`-Vertrag (list/start/stop/create + Capability-Flags), je Provider **ein SDK-freier REST-Adapter** über eingebautes `fetch` (`src/vps/VpsProviderRegistry.js` + `src/vps/providers/<provider>.js`). Endpunkte `GET /api/vps/providers`, `GET /api/vps/machines`, `POST /api/vps/machines/{provider}` (Create-from-scratch mit cloud-init-user-data), `POST /api/vps/machines/{provider}/{serverId}/start|stop`. Provider-API-Token store-intern aus dem `CredentialStore` (`vps/<provider>_api_token`), transient pro Aufruf, **nie** in Response/Log/Audit/WS/Argv. Read-Aggregation **live** (kein Maschinen-Store, ADR-005-Linie), pro Provider degradierend. Create übergibt cloud-init-user-data (erzeugt von der separaten `CloudInitBuilder`-Komponente, [[vps-cloud-init-setup]]) + root-/alex-Public-Keys ([[vps-ssh-key-assignment]]). Normalisiertes Maschinen-Modell `VpsMachine` (fehlende Felder → `null`/`unknown`, nie Fehler). Server-mutierende Aktionen auditiert (Audit-First) + identitäts-/rollengeschützt (gleiche `CRED_ADMIN_EMAILS`-Logik wie ADR-007). **Detail-Festlegung (SDK-vs-REST je Provider, Adapter-Layout, Token-Injektion, cloud-init-Owner, Normalisierung): ADR-009.** Specs: [[vps-provider-boundary]], [[vps-cloud-init-setup]], [[vps-ssh-key-assignment]], [[view-vps]]; **Rebuild/Backup vertagt** ([[vps-rebuild-backup]], Platzhalter).

**Davor (Infra, nicht im Image):** Cloudflare **Access** (Identitäts-Gate) + Tunnel-Route `devgui.<domain>`.

## Kern-Flows
1. **Flow auslösen:** Panel-Klick → `POST /api/command {command}` → AccessGuard ok → CommandService: Allowlist ok? Lock frei? → schreibt `command\n` in den PTY, Command→`running`, AuditEntry → PTY-Output streamt via WS → xterm rendert → bei Prompt-Ende Command→`done`, Lock frei.
2. **Status laden:** Frontend abonniert `GET /api/status` (Polling/SSE) → GitHubReader + DockerReader aggregieren **live** → Dashboard rendert.
3. **Kill-Switch:** `POST /api/command/cancel` → Interrupt (Ctrl-C) an die Session → Command→`cancelled`, Lock frei.
4. **Session-Bootstrap:** Container-Start → `claude` per **Abo-OAuth** (persistierte Credentials, kein API-Key) interaktiv im PTY → Session `ready`.
5. **Cloudflare-Inventar (A):** Cloudflare-View → `GET /api/cloudflare/zones` (live) → Zone anwählen → `GET /api/cloudflare/zones/{zoneId}/tunnels` (Tunnel + Public-Hostname-Routen, jede Route mit `protected`-Flag aus LockoutGuard) → Lösch-Klick auf nicht-protected Route → type-to-confirm (Hostname tippen) → `DELETE /api/cloudflare/tunnels/{tunnelId}/routes/{hostname}` (+ ggf. DNS-Record) → CloudflareApi: LockoutGuard-Check (hart) → Audit-First → Mutation → Re-Fetch.
6. **Deploy-Lifecycle (B):** Deployments-View → `POST /api/deployments {image, vps, hostname}` → DeployOrchestrator: (a) `VpsDockerControl` pullt+startet Container mit Label `cloudflare.tunnel-hostname=<hostname>`, (b) `CloudflareApi` legt Tunnel-Route (`PUT …/configurations`) + DNS-CNAME an. Schlägt (b) fehl → (a) wird rückabgewickelt (Container `rm`) → kein verwaister Container. Undeploy = inverse Einheit (Route+DNS entfernen, dann Container `rm`); protected Hostname → 422, kein Undeploy.
7. **Reconciliation (C, beidseitig selbst-heilend):** Mitternachts-Tick (oder manueller `POST /api/deployments/reconcile`) → ReconciliationJob iteriert je konfiguriertem VPS: `VpsDockerControl` liest laufende Container-Labels (`cloudflare.tunnel-hostname`), `CloudflareApi` liest die Tunnel-Routen → Routen **ohne** managed Container UND **nicht** protected → entfernen (Audit-First); **managed** Container **ohne** Route → Route **anlegen** über den atomaren Anlege-Pfad des `DeployOrchestrator` (ADR-012, nie protected). Jede Aktion → `ReconcileNotice` (im Cloudflare-Panel sichtbar); jeder Lauf → `ReconcileReport`, beide über `AuditStore` persistiert. Degradiert pro VPS/Provider; ein Ausfall kippt den Lauf nicht; fail-closed bei unsicherem Container-Bestand.

## Zustände
- **Session:** `starting → ready ⇄ busy`; `→ stopped` (down), `→ failed` (Restart-Limit überschritten). Restart-Policy: bis *N* Neustarts in *M* s, sonst `failed`.
- **Command:** `queued → running → (done | cancelled)`. Globaler Lock: max **1** `running`.

## Externe Schnittstellen
- **Claude Code CLI** — interaktiv via PTY (`node-pty`). **Vertrag:** Start **ohne** `-p`/`--print`, **ohne** `ANTHROPIC_API_KEY`; Auth = Abo-OAuth (persistierte Credentials). Pre-granted Tool-Permissions (dokumentiert).
- **GitHub** — REST/GraphQL via App-Token: read-only über `GitHubReader`; **schreibend** über `GitHubWriter` (Repo anlegen via REST, Repo klonen via `git clone` in `WORKSPACE_DIR`). Installation-Token (~1h TTL) transient pro Aufruf gemintet.
- **Workspace (Dateisystem)** — der wirksame Workspace-Root ist der **Effektivwert** `konfiguriert ?? Env WORKSPACE_DIR` ([[workspace-path-config]]) auf dem persistenten Volume, uid-1000-schreibbar, beim Boot idempotent angelegt: Ziel lokaler Repo-Klone ([[github-repo-clone]]) und Quelle der Workspace-Übersicht ([[workspace-repos]]: Listing/Pull/Löschen). Der über die Settings-Ansicht **konfigurierbare** Root muss **innerhalb** der gemounteten Schranke `WORKSPACE_DIR` liegen (Container-Mount-Grenze, Modell a — Path-Traversal-/Symlink-sicher); ist nichts konfiguriert, gilt die Env unverändert. Der konfigurierte Pfad ist **nicht-geheime** Betreiber-Konfiguration → Klartext-Metadatum (z.B. `meta`-Block des `CredentialStore`), **nicht** im verschlüsselten `entries`-Block. Alle Klon-/Pull-/Lösch-Ziele strikt innerhalb des Effektivwerts; die drei Workspace-Boundaries (`GitHubCloner`/`WorkspaceScanner`/`WorkspaceMutator`) lesen den Effektivwert über **eine** gemeinsame Auflösung statt direkt `process.env.WORKSPACE_DIR`. Tiefe Architektur (Resolver-Komponente, wo der Pfad persistiert) = `architekt`.
- **Docker Engine** — Socket, read-only Nutzung (`ps`/`inspect`).
- **Cloudflare Access** — JWT im Header `Cf-Access-Jwt-Assertion`; validiert gegen Team-Domain + AUD (Public Keys von `/cdn-cgi/access/certs`).
- **Cloudflare API** — REST/JSON via Bearer-Token über eingebautes `fetch` (kein SDK), nur über `CloudflareApi`: Zones (`GET /zones`), DNS-Records (`GET/POST/DELETE /zones/{id}/dns_records`), **remote-managed Tunnels** (`GET/PUT /accounts/{accId}/cfd_tunnel/{id}/configurations`, `GET /accounts/{accId}/cfd_tunnel`, `DELETE …/{id}`). Token + Account-Id store-intern aus dem `CredentialStore` (transient, nie geleakt). Lese-Status live (ADR-005-Linie), pro Zone degradierend. **Selbstbezug:** `devgui`-Route + Access-Mauer-Records sind über `LockoutGuard` (ADR-011) hart vor Mutation geschützt.
- **VPS (Docker, schreibend, via SSH)** — `docker pull/run/rm` auf einem laufenden VPS über die `VpsProvisioner`-SSH-Linie (ADR-008); nur über `VpsDockerControl`. Container tragen das Label `cloudflare.tunnel-hostname=<app>.<domain>` als einzige (store-lose) Container↔Route-Bindung. Read-only Docker bleibt lokal (`DockerReader`).

## NFRs (prüfbar)
- **Sicherheit (Floor):** keine `/api/*`-/WS-Anfrage ohne gültigen Access-Nachweis (→403). Dienst **startet nicht** ohne Access-Konfig in Produktion. Keine Secrets (API-Key, GPG-Passphrase, Tokens) in Frontend-Bundle, Logs, Audit oder WS-Stream.
- **Kosten:** Engine nutzt ausschließlich die interaktive Abo-Session — **kein** API/`-p`. (Testbar: Prozess-Argv + Environment.)
- **Concurrency:** global **max. 1** laufender Command.
- **Beobachtbarkeit:** jeder ausgelöste Befehl im append-only Audit-Log (`GET /api/audit`).
- **Verfügbarkeit:** Session überlebt Absturz via Restart-Policy.

## Entscheidungen (ADR)
- **ADR-001 · 2026-05-26 · Engine = interaktive Claude-Session via `node-pty`** (nicht Agent-SDK, nicht `claude -p`). *Grund:* Abo deckt nur interaktive Nutzung kostenlos; API + `-p` kosten Token bzw. ziehen ab 2026-06-15 aus separatem SDK-Kontingent. *Verworfen:* Agent-SDK (`@anthropic-ai/claude-agent-sdk`, API-Cost), `claude -p` (separates Kontingent).
- **ADR-002 · 2026-05-26 · Session-Ort = VPS, always-on.** *Grund:* jederzeit über `devgui` erreichbar. *Verworfen:* Mac-Runner (muss laufen) — geringere Cred-Fläche, aber nicht always-on.
- **ADR-003 · 2026-05-26 · Pre-granted, unbeaufsichtigt.** *Grund:* maximaler Komfort, Voll-Steuerung ab Tag 1. *Verworfen:* attended (Genehmigung pro Prompt). *Risiko* (RCE-Fläche) bewusst getragen — kompensiert durch ADR-004 + Leitplanken (`access-and-guardrails`).
- **ADR-004 · 2026-05-26 · Auth = Cloudflare Access** (kein App-Login). *Grund:* Zero-Trust ohne App-Code; einzige Mauer vor der pre-granted Engine. *Verworfen:* Supabase-JWT (mehr Code, weiterer Dienst).
- **ADR-005 · 2026-05-26 · Kein eigener State-Store.** GitHub + Docker sind Source of Truth; jede Statusantwort wird live ermittelt.
- **ADR-006 · 2026-05-26 · Stack = Node-Vollstack** (React + Express + ws + `node-pty` + xterm.js). *Grund:* ein Dienst, SDK-frei, passt zur PTY-Bridge.
- **ADR-007 · 2026-06-07 · ENTSCHIEDEN · Credential-Store = verschlüsselte JSON-Datei auf dem persistenten Volume (AES-256-GCM, Master-Key aus Boot-Env).** Siehe Vollfassung unten.
- **ADR-008 · 2026-06-07 · ENTSCHIEDEN (Skizze) · VPS-/SSH-Provisionierung = eigener `VpsProvisioner`-Boundary, SSH-from-Backend.** Boundary-Festlegung, damit #45/#46 nicht in eine Sackgasse bauen. **ERWEITERT 2026-06-08 (Key-Rotation, #117/#118):** Der `VpsProvisioner` bekommt zwei neue primitive SSH-Operationen — `removeAuthorizedKey` (gezieltes idempotentes Entfernen EINES Public-Keys) + `testConnection` (Login-Probe mit einem bestimmten Private-Key); die Rotation-**Orchestrierung** (gen→add→test→remove, Rollback) liegt **außerhalb** in einer aufrufenden Schicht (#118), nicht im Boundary. Siehe Vollfassung + Erweiterungs-Abschnitt unten. Specs: [[ssh-key-rotation]], [[ssh-key-generation]].
- **ADR-009 · 2026-06-08 · ENTSCHIEDEN · Multi-Provider-VPS = ein `VpsProviderRegistry`-Boundary mit drei SDK-freien REST-Adaptern (Hetzner, IONOS, Hostinger) über eingebautes `fetch`; provider-agnostischer `VpsProvider`-Vertrag + Capability-Flags; cloud-init-Erzeugung als eigene `CloudInitBuilder`-Komponente; Provider-Token transient store-intern aus dem `CredentialStore`.** Read-Aggregation live (ADR-005-Linie), pro Provider degradierend; Rebuild/Backup bewusst vertagt. Siehe Vollfassung unten.
- **ADR-010 · 2026-06-08 · ENTSCHIEDEN · Cloudflare-API-Boundary = ein `CloudflareApi`-Boundary, SDK-frei via eingebautes `fetch`; remote-managed Tunnels (Cloudflare-API verwaltet die Ingress-Config), kein SSH für Routen-Mutation.** Token + Account-Id transient store-intern aus dem `CredentialStore` (`credentials/cloudflare/*`); Read live (ADR-005-Linie), pro Zone degradierend. Migrationspfad vom heutigen local-config-Modell benannt. Siehe Vollfassung unten.
- **ADR-011 · 2026-06-08 · ENTSCHIEDEN · Self-Lockout-Schutz = `LockoutGuard`-Hard-Block + type-to-confirm.** Routen/Records der eigenen `devgui`-Erreichbarkeit ODER der Access-Mauer sind per Erkennung **hart gesperrt** (keine Mutation, egal welche Identität/welcher Token); jede andere Tunnel-/Routen-Löschung erfordert einen type-to-confirm-Bestätigungstoken (Hostname tippen) — analog der Rebuild-Guardrail in ADR-009. Siehe Vollfassung unten.
- **ADR-012 · 2026-06-08 · ENTSCHIEDEN · Deploy-Lifecycle = eigener `DeployOrchestrator`-Boundary (Container+Route als atomare Einheit) + neue `VpsDockerControl`-Boundary (Docker-on-VPS via SSH, ADR-008-Linie).** Container↔Route-Bindung store-los über das Container-Label `cloudflare.tunnel-hostname`. Nicht in die Cloudflare-View gedrängt (eigene Deployments-View/Route). Siehe Vollfassung unten.

### ADR-007 (Vollfassung) · Verschlüsselter Credential-Store

**Status:** ENTSCHIEDEN · 2026-06-07 · Entscheider: `architekt` · betrifft `settings-credentials` (#45), `settings-ssh-keys` Stufe A (#46).

**Kontext.** Die Settings-Ansicht muss Geheimnisse (Integration-Credentials + optionale SSH-Private-Keys) **persistent** und **at rest verschlüsselt** ablegen. Das kollidiert nominal mit ADR-005/Nicht-Ziel „keine eigene DB". Rahmen: Non-Root-Container (uid 1000), genau **ein** persistentes Named-Volume (`dev-gui-claude` → `/home/node/.claude`), Cloudflare Access davor, ein bis zwei Identitäten, write-only Richtung Frontend, kein Secret in Code/Log. Im Projektumfeld existiert bereits das Muster „verschlüsselte Datei + Passphrase aus Env" (agent-flow `.env.gpg` + `GPG_PASSPHRASE`).

**Betrachtete Optionen.**
1. **Verschlüsselte JSON-Datei auf dem persistenten Volume**, App-eigene AES-256-GCM-Krypto via Node-`crypto`, Master-Key aus Boot-Env. *(gewählt)*
2. GPG-Datei analog `.env.gpg` (Reuse der vorhandenen GPG-Mechanik). *Verworfen:* GPG ist für **statische, beim Deploy einmal verschlüsselte** Env-Dateien gedacht; pro Mutation einen `gpg`-Subprozess zu spawnen (write-many) ist umständlich, schwer atomar und vermischt Geheimnis-Klartext mit Prozess-Argv/Tempfiles. Die Passphrase-aus-Env-**Idee** wird aber übernommen.
3. Externer Secret-Manager (Vault/Cloud-KMS). *Verworfen:* zusätzlicher Dienst + Netz-/Auth-Abhängigkeit, überdimensioniert für 1–2 Identitäten; widerspricht „ein Dienst, SDK-frei" (ADR-006). Bleibt als spätere Migration offen (Boundary kapselt das, s.u.).
4. Container-Secret (Docker/Compose secret). *Verworfen:* Docker-Secrets sind **read-only zur Laufzeit** und beim Deploy gesetzt — taugen nicht für ein zur Laufzeit über die GUI **beschreibbares** Store.

**Entscheidung.**
- **Speicherort:** eine Datei `secrets.enc.json` unter `/home/node/.claude/dev-gui/` (also auf dem bestehenden Volume — kein neues Volume, keine echte DB; bewusste, eng begrenzte Erweiterung von ADR-005, s. „Verhältnis zu ADR-005"). Verzeichnis beim Boot idempotent anlegen (uid-1000-owned, mode `0700`); Datei mode `0600`.
- **Krypto:** **AES-256-GCM** via Node-Builtin `crypto` (kein Drittlib). Pro Eintrag (oder pro Datei-Schreibvorgang) ein frischer 12-Byte-Random-IV; GCM-Auth-Tag wird mitgespeichert und bei Lesen verifiziert (Integritätsschutz → manipulierte Datei = harter Fehler, kein stilles Fallback).
- **Datei-Schema (kanonisch):**
  ```json
  {
    "version": 1,
    "kdf": { "algo": "scrypt", "salt": "<base64>", "N": 16384, "r": 8, "p": 1 },
    "entries": {
      "credentials/<integration>/<name>": { "iv": "<b64>", "tag": "<b64>", "ct": "<b64>", "updatedAt": "<iso>" },
      "ssh/<user>/private_key":          { "iv": "<b64>", "tag": "<b64>", "ct": "<b64>", "updatedAt": "<iso>" }
    }
  }
  ```
  Public-Keys sind **nicht geheim** → sie gehören **nicht** verschlüsselt in `entries`, sondern als Klartext-Metadatum (gleiche Datei, eigener `meta`-Block, oder separate `meta.json`); `coder` wählt die einfachste Variante, solange Public-Keys getrennt vom verschlüsselten Block liegen.
- **Master-Key-Herkunft (Boot):** Env-Var **`CRED_MASTER_KEY`** (Compose-`environment`, aus `.env`/Host-Secret — **nie** im Image, nie geloggt). Daraus wird mit **scrypt** (Salt aus der Datei, beim ersten Schreiben generiert) der 32-Byte-AES-Schlüssel abgeleitet — der Roh-Env-Wert wird **nie** direkt als Key benutzt und **nie** persistiert. Fehlt `CRED_MASTER_KEY` in Produktion, aber `secrets.enc.json` enthält **verschlüsselte Einträge** (d.h. `kdf`-Block vorhanden UND `entries` nicht leer) → **Fail-Fast beim Boot** (analog `assertAccessConfig`), damit nicht stillschweigend ein leeres/neues Store entsteht. Eine reine Meta-Datei ohne verschlüsselte Einträge (z.B. nur Public-Keys im `meta`-Block) löst diesen Fail-Fast **nicht** aus — sie enthält keine Geheimnisse und erfordert keinen Key. Optionaler Datei-Fallback `CRED_MASTER_KEY_FILE` (Pfad, Wert wird gelesen) analog zur `GPG_PASSPHRASE`-Option-B im Compose — `coder` darf das ergänzen, Env hat Vorrang.
- **Schreibsicherheit (atomar):** Mutationen schreiben in `secrets.enc.json.tmp` + `fsync` + `rename()` → kein halb geschriebenes Store bei Crash. In-Memory wird die **entschlüsselte** Form nur transient pro Request gehalten, nie über den Request hinaus gecached (Vereinfachung; bei 1–2 Identitäten unkritisch). Concurrency: ein prozess-interner Schreib-Mutex (kleine Schreiblast).
- **Boundary:** genau **eine** Komponente **`CredentialStore`** (`src/CredentialStore.js`) ist einziger Lese-/Schreibpfad zu `secrets.enc.json`. Vertrag (write-only nach außen): `get(key)`/`list()` liefern an HTTP-Handler **nur Metadaten** (`status`, optional Maske, `updatedAt`) — Klartext verlässt den Store **ausschließlich** intern an Konsumenten (z.B. künftiger `VpsProvisioner`), **nie** in eine HTTP-Response, Log, Audit oder WS-Stream. Maske = höchstens letzte 4 Zeichen (für kurze Werte: nur „•••• gesetzt").
- **Rollenfrage (OA3):** **Gültige Cloudflare-Access-Identität genügt** zum Mutieren — **plus** zwingender Audit-Eintrag pro Mutation. *Begründung:* privates Werkzeug für 1–2 vorab durch Access freigeschaltete Identitäten; eine zweite App-interne Allowlist wäre Doppel-Gate ohne Mehrwert (ADR-004-Linie: Access ist die Mauer). **Erweiterungspunkt (prüfbar vorbereitet):** eine optionale Env **`CRED_ADMIN_EMAILS`** (Komma-Liste). Ist sie **gesetzt**, dürfen nur diese `req.identity.email` mutieren (sonst 403); ist sie **leer/ungesetzt**, gilt „jede gültige Access-Identität". Damit ist AC7 in beiden Auslegungen erfüllbar und eine spätere Rollentrennung ohne Re-Architektur möglich. Die Identität kommt aus dem bestehenden `req.identity.email` (AccessGuard).

**Verhältnis zu ADR-005 / Re-Scoping.** ADR-005 bleibt unverändert gültig für **Fabrik-Read-Models** (Project/Board/CI/Preview = live aus GitHub+Docker, kein Store). ADR-007 ergänzt eine **eng abgegrenzte, verschlüsselte Konfig-Ablage** ausschließlich für **vom Betreiber gesetzte Geheimnisse**. Re-Scoping des Nicht-Ziels: „keine eigene DB für **Fabrik-/Domänen-State**" — Betreiber-**Konfiguration/Credentials** sind davon ausgenommen und werden als verschlüsselte Datei (kein RDBMS, kein Query-Layer) auf dem ohnehin vorhandenen Volume gehalten. Kein neuer Dienst, keine DB-Engine → ADR-006 („ein Dienst, SDK-frei") bleibt gewahrt.

**Konsequenzen (prüfbar — Review-Kriterien).**
- `CredentialStore` ist der **einzige** Modul, der `secrets.enc.json` öffnet (Grep: kein anderer `fs`-Zugriff auf den Pfad). *(architecture/R01)*
- Keine HTTP-Response / kein Log / kein Audit / kein WS-Frame enthält je einen Geheimnis-Klartext (testbar: Response-Body/Audit enthalten gesetzten Wert nicht — AC4/AC6).
- `CRED_MASTER_KEY` erscheint nicht im Image (Grep Dockerfile), nicht im Log, nicht im Frontend-Bundle.
- Produktionsstart ohne `CRED_MASTER_KEY`, wenn `secrets.enc.json` verschlüsselte Einträge enthält (`kdf`-Block vorhanden UND `entries` nicht leer) → Prozess bricht ab (Fail-Fast), analog AccessGuard. Reine Meta-Datei (nur `meta`-Block, keine verschlüsselten `entries`) löst keinen Fail-Fast aus.
- Manipuliertes/inkonsistentes Store (GCM-Tag falsch) → harter Fehler, **kein** stiller Reset, kein Teil-/Klartext-Leak (Spec-Edge-Case „Backend nicht erreichbar/konsistent → 5xx ohne Leak").
- Jede Mutation erzeugt vor Ausführung einen `AuditStore`-Eintrag (Identität, Feld-Key, Aktion) **ohne** Klartext; schlägt der Audit-Write fehl → Mutation unterbleibt (bestehende AuditStore-Vertragslogik).
- Neue Env-Vars für `coder`/Deployment: `CRED_MASTER_KEY` (Pflicht in Prod), optional `CRED_MASTER_KEY_FILE`, optional `CRED_ADMIN_EMAILS`. In `docker-compose.yml` als `environment`-Einträge zu ergänzen (auskommentierter Block analog `GPG_PASSPHRASE`).

**Tradeoff / Ausnahme (ssh-key-generation #115).** Das write-only-Prinzip „Klartext verlässt den Store nie in HTTP-Response/Log/Audit/WS" wird **bewusst und dauerhaft** für VPS-SSH-Private-Keys gebrochen: der Export-Endpunkt `GET /api/settings/ssh-keys/{user}/private-key/export` liefert den Private-Key-Klartext als HTTP-Response. Dies ist die **einzige** Ausnahme; alle anderen Pfade (normale Lese-Antworten, Logs, Audit, WS-Stream, Frontend-Bundle) liefern weiterhin **nie** Private-Key-Klartext. Begründung und Einhegungen sind in `docs/specs/ssh-key-generation.md` (Abschnitt „Sicherheits-Tradeoff") vollständig beschrieben; zusammengefasst gelten drei Einhegungen: (1) **Access-Mauer** (Cloudflare Access vor dem Endpunkt), (2) zusätzlicher **`CRED_ADMIN_EMAILS`-Rollencheck** (gleiche Logik wie ADR-007 OA3) und (3) **Audit-First bei jedem Export** — vor der Auslieferung wird ein Audit-Eintrag (Identität, Rollen-Label, Aktion `ssh-key-export`, Zeit) **ohne** Key-Klartext geschrieben; schlägt der Audit-Write fehl, unterbleibt der Export.

### ADR-008 (Skizze) · VPS-/SSH-Provisionierungs-Boundary

**Status:** ENTSCHIEDEN als **Boundary-Festlegung** (Skizze) · 2026-06-07 · betrifft `settings-ssh-keys` Stufe B (#47) + `view-vps`. Zweck: verhindern, dass #45/#46 in eine Sackgasse bauen. Die **Detail-Spec** der Provisionierung bleibt Folge-Anforderung; hier wird nur die Architektur-Grenze fixiert, an die #46 andocken darf.

**Kontext.** Stufe B (`POST /api/settings/ssh-keys/{user}/provision`) soll einen hinterlegten **Public-Key idempotent** in `authorized_keys` eines VPS-Ziels eintragen. Das ist eine **schreibende externe Integration** (neuer Boundary). #46 (Stufe A: Key-Verwaltung) darf jetzt nicht so bauen, dass Stufe B später kollidiert.

**Entscheidung (Boundary, bindend).**
- **Mechanik:** **SSH-from-Backend** — der Dienst verbindet sich als SSH-Client direkt zum VPS und schreibt idempotent in `authorized_keys` (z.B. mit `ssh-copy-id`-Semantik bzw. append-if-absent). *Verworfen:* Hetzner-Provider-API für Key-Injection — die Provider-Cloud-Init-API setzt Keys nur **bei Server-Erstellung**, nicht idempotent auf laufende Server; SSH ist provider-unabhängig und deckt alle Ziel-Benutzer (`root`, `alex`) ab.
- **Authentifizierungs-Quelle:** der für die SSH-Verbindung nötige **Private-Key** stammt aus dem **`CredentialStore` (ADR-007)** — Stufe A legt ihn dort write-only ab; Stufe B konsumiert ihn **store-intern** (Klartext verlässt den Store nie Richtung HTTP/Log). Damit ist Stufe A → Stufe B über genau eine Boundary verbunden.
- **Neue Komponente (später):** **`VpsProvisioner`** (`src/VpsProvisioner.js`) — einziger Ort, der SSH-Verbindungen aufbaut. Konsumiert `CredentialStore` (Private-Key) + ein **VPS-Ziel-Schema**.
- **VPS-Ziel-Schema (vorab fixiert, damit #46 stabil bleibt):** ein Ziel ist `{ host, port?: 22, targetUser }` (frei eintragbar/„misc"-artig; Hetzner-spezifische Server-Auflösung kann später ergänzt werden). `view-vps` (#48+) liefert später eine Auswahl; bis dahin genügt direkte Eingabe der Ziel-Referenz im Provision-Request.
- **Schutz:** hoch-privilegiert → Access-Mauer + Audit + identitäts-/rollengeschützt (gleiche `CRED_ADMIN_EMAILS`-Logik wie ADR-007), idempotent, explizites `{result, reason?}` ohne Geheim-Leak (AC7–AC10).

**Was #45/#46 jetzt einhalten müssen (Sackgassen-Vermeidung — prüfbar).**
- #46 legt Private-Keys **ausschließlich** über `CredentialStore` (ADR-007) ab — **kein** separater SSH-Key-Store, kein Schreiben in das echte `~/.ssh/` des Containers. Schlüssel-Key-Schema: `ssh/<user>/private_key` (Geheimnis) + Public-Key als Klartext-Metadatum (s. ADR-007-Schema).
- #45/#46 führen **keinen** SSH-Client/Provider-Code ein — der bleibt allein in der späteren `VpsProvisioner`-Boundary.
- Die Provision-Route existiert in #46 (Stufe A) **nicht** oder nur als 501/„not yet" — Detail-Implementierung = #47.

**Konsequenz / offen für die Folge-Spec.** Wahl der SSH-Lib (Node `ssh2` vs. Spawn von System-`ssh`/`ssh-copy-id`), Host-Key-Verifikation (known_hosts-Strategie) und das endgültige `view-vps`-Ziel-Schema werden in der #47-Spec entschieden — sie berühren keine #45/#46-Verträge mehr.

**Umsetzungsstand (#47 Done — bindend für die Erweiterung).** `src/VpsProvisioner.js` ist umgesetzt: Transport = Node-Lib **`ssh2`** (kein Shell-Out, kein System-`ssh`; das Image enthält keinen openssh-client). Provision schreibt idempotent über eine via stdin an `bash -s` übergebene Shell-Logik (`mkdir -p ~/.ssh` `700` → `touch authorized_keys` `600` → `grep -qF`-Exaktmatch → `>>` append). Host-Key-Policy: SHA256-Fingerprint (Base64) im `hostVerifier`; mit `hostFingerprint` strenge Prüfung (Mismatch → `host-key-mismatch`), ohne `hostFingerprint` TOFU-accept + Hash im Audit. Connect-Timeout 15 s. Fehlerklassen `no-public-key | no-private-key | unreachable | auth-failed | host-key-mismatch | error`. Diese Mechanik ist die **Basis**, an der die Key-Rotation-Erweiterung (s.u.) andockt.

### ADR-008-Erweiterung (Key-Rotation) · 2026-06-08

**Status:** ENTSCHIEDEN · 2026-06-08 · Entscheider: `architekt` · betrifft [[ssh-key-rotation]] (AC1–AC8), [[ssh-key-generation]] (Key-Quelle). Baut auf der umgesetzten #47-Basis (`src/VpsProvisioner.js`, `ssh2`, TOFU/`hostFingerprint`, Audit-First) + ADR-007 (`CredentialStore`, `CRED_ADMIN_EMAILS`). Umsetzungs-Items: **#117** (Provisioner-Erweiterung) + **#118** (Rotation-Orchestrierung). **Ändert die #47-Provision-Mechanik nicht** — ergänzt nur zwei primitive Operationen am selben Boundary.

**Kontext.** Die vollautomatische, additive, rollback-sichere SSH-Key-Rotation ([[ssh-key-rotation]]) braucht über das bisherige „Public-Key idempotent eintragen" hinaus **zwei** neue SSH-Primitive: (1) einen **bestimmten** alten Public-Key gezielt aus `authorized_keys` der Ziel-Rolle entfernen, ohne andere Keys zu beschädigen, und (2) einen **Login-Test mit dem neuen Private-Key**, der *vor* dem Entfernen des alten Keys beweist, dass der neue Key Zugang gibt (der Verbindungstest **ist** der Aussperr-Schutz). Beide bleiben im selben **SSH-from-Backend-Boundary** (`VpsProvisioner` ist und bleibt der **einzige** Ort, der SSH-Verbindungen aufbaut, ADR-008). Der für den Test nötige neue Private-Key stammt store-intern aus dem `CredentialStore` (ADR-007) und verlässt das Backend **nie** Richtung HTTP/Log/Audit/WS.

**Betrachtete Optionen.**

*Option-Achse 1 — Matching-Strategie für `removeAuthorizedKey` (die prägende Weiche):*
1. **Exaktes Matching über den normalisierten Key-Blob (`<type> <base64>`), zeilenweise, options-/kommentar-tolerant; Datei serverseitig neu schreiben (read → filter → atomic write).** *(gewählt)* Der Public-Key besteht aus `[options] <type> <base64-blob> [comment]`. Die **stabile, eindeutige Identität** ist das Paar `(<type>, <base64-blob>)` — Kommentar und etwaige Options-Prefixe sind frei veränderbar und dürfen das Matching **nicht** bestimmen. Pro `authorized_keys`-Zeile wird der Key-Blob extrahiert (erstes `ssh-…`/`ecdsa-…`-Token + folgendes base64-Token), per Whitespace normalisiert und gegen den Blob des zu entfernenden Keys verglichen; **alle** Zeilen mit diesem Blob werden entfernt (Duplikat-tolerant), alle anderen Zeilen bleiben **bytegenau** erhalten. Geschrieben wird die gefilterte Datei **atomar** (Tempfile im selben `.ssh`-Verzeichnis + `chmod 600` + `mv` über das Original) — kein In-place-`sed`, kein halb geschriebenes `authorized_keys`. Idempotent: Blob nicht vorhanden → Datei unverändert, `result: "already-absent"`.
2. **Match nur über die ganze, byteidentische Key-Zeile.** *Verworfen:* bricht, sobald der alte Key mit abweichendem Kommentar/Whitespace/Options-Prefix eingetragen wurde (z.B. von cloud-init vs. von `addAuthorizedKey`) — dann bliebe der alte Key trotz „Erfolg" stehen → Sicherheitslücke (alter Key weiter gültig). Whitespace-/Kommentar-Drift ist im realen `authorized_keys` der Normalfall.
3. **Match über den Kommentar (das `comment`-Feld der Key-Zeile).** *Verworfen:* Kommentare sind nicht eindeutig, frei wählbar und können fehlen; ein Kommentar-Match könnte den **falschen** Key entfernen (Aussperr-Risiko) oder einen umbenannten Key verfehlen.
4. **In-place `sed -i '/.../d'` auf dem Server.** *Verworfen:* der base64-Blob enthält Regex-Metazeichen (`/`, `+`) → Escaping-Falle/Injection-Fläche; `sed -i` ist auf vielen Systemen nicht atomar (Truncate-Fenster) → ein Crash mitten im Schreiben kann `authorized_keys` beschädigen und **alle** Keys löschen (Total-Lockout). Das read→filter→atomic-write-Muster ist robuster und hält die single-quote-Escape-Linie der #47-Provision-Logik ein.

*Option-Achse 2 — Was `testConnection` genau prüft:*
5. **Erfolgreicher SSH-Auth-Handshake (`ready`) + ein harmloses `true`-Kommando (exit 0).** *(gewählt)* Der `ready`-Event von `ssh2` feuert erst **nach** erfolgreicher Authentifizierung mit dem übergebenen Private-Key — das ist bereits der eigentliche Zugangs-Beweis. Zur Absicherung gegen Auth-Akzeptanz-ohne-Shell wird zusätzlich ein **no-op `true`** ausgeführt und auf `exit code 0` geprüft. Erst `ready` **und** `exit 0` → `{ ok: true }`. Alles andere (Connect-Fehler, Auth-Reject, Host-Key-Mismatch, Timeout, Non-Zero-Exit) → `{ ok: false, reason }` mit derselben Fehlerklassifikation wie #47.
6. **Nur TCP-/Port-Reachability prüfen.** *Verworfen:* ein offener Port beweist **nicht**, dass der **neue Key** authentifiziert — das ist genau die false-positive-Falle, die den alten Key fälschlich freigeben würde. Der Test **muss** die Key-Authentifizierung selbst durchlaufen.
7. **Auth-Reject als „Test grün" tolerieren / Fehler verschlucken.** *Verworfen:* würde den Aussperr-Schutz aushebeln. **Fehlschlag MUSS eindeutig `ok: false` sein**, niemals ein false-positive: jeder nicht eindeutig als „authentifiziert + exit 0" verifizierte Ausgang ist `ok: false`.

**Entscheidung (bindend).**

- **Zwei neue primitive Operationen am `VpsProvisioner` (einziger SSH-from-Backend-Ort, ADR-008).** Der Boundary bekommt **nur** diese zwei Primitive; er kennt **keine** Rotation-Reihenfolge und **keinen** Rollback (s. Orchestrierungs-Grenze). Kanonische Signaturen (konsistent mit dem ziel-zentrierten Schema der [[ssh-key-rotation]]-Verträge, `{ host, port?, targetUser, … }`):
  ```
  removeAuthorizedKey({ host, port?, targetUser, publicKey, hostFingerprint? })
      → { result: "removed" | "already-absent" | "error", reason?, errorClass?, hostKeyHash? }

  testConnection({ host, port?, targetUser, privateKey, hostFingerprint? })
      → { ok: boolean, reason?, errorClass?, hostKeyHash? }
  ```
  - `removeAuthorizedKey`: matcht den **normalisierten Key-Blob** (`<type> <base64>`) des übergebenen `publicKey`, entfernt **alle** passenden Zeilen, lässt jede andere Zeile bytegenau stehen, schreibt `authorized_keys` **atomar** neu (Tempfile + `chmod 600` + `mv`), idempotent (`already-absent` wenn nicht vorhanden). Verändert weder `.ssh`-Mode noch andere Dateien.
  - `testConnection`: baut die SSH-Verbindung mit dem übergebenen `privateKey` auf, verifiziert `ready` (= Auth ok) **und** führt `true` aus (exit 0). Kein Schreibzugriff, keine Mutation der Ziel-Datei. `privateKey` kommt store-intern (vom Aufrufer aus dem `CredentialStore` gezogen) und wird **nie** geloggt/auditiert/zurückgegeben.
  - **Host-Key-Strategie identisch zu #47:** SHA256-Fingerprint im `hostVerifier`; mit `hostFingerprint` strenge Prüfung (Mismatch → `errorClass: "host-key-mismatch"`), ohne `hostFingerprint` TOFU-accept + `hostKeyHash` im Ergebnis für den Audit. **Connect-/Auth-Timeout = derselbe `CONNECT_TIMEOUT_MS` (15 s)** wie #47 — keine zweite, abweichende Timeout-Quelle.
  - **Verhältnis zur bestehenden #47-Methode.** Die Rotation-Spec nennt das additive Eintragen `addAuthorizedKey(...)`; der heutige Code exponiert es als `provision(sshUser, target, opts)` (zieht Public+Private selbst aus dem Store). Bindend für #117: das **additive Einspielen, das Entfernen und der Test** werden als **drei gleichrangige, ziel-zentrierte Primitive** angeboten, die den **publicKey bzw. privateKey als Argument** entgegennehmen (nicht selbst aus dem Store ziehen) — denn die Orchestrierung (#118) braucht in einem Lauf den **neuen** und den **alten** Key gleichzeitig (additiv neu, später alt entfernen), die nicht beide unter `ssh/<user>/private_key` liegen. `coder` darf die bestehende store-ziehende `provision(...)` als dünnen Komfort-Wrapper über das key-als-Argument-Primitiv erhalten (Rückwärtskompatibilität der #47-Route), **muss** aber das key-als-Argument-Primitiv `addAuthorizedKey({ …, publicKey, … })` bereitstellen, gegen das #118 baut. Alle drei Primitive teilen die identische Connect-/Host-Key-/Fehler-/Timeout-Mechanik (eine `connect`-Hilfsfunktion, kein zweiter SSH-Pfad).

- **Orchestrierungs-Grenze (bindend — #117 vs. #118).** Die Rotation-**Orchestrierung** (Reihenfolge gen → additiv einspielen → Test → bei grünem Test alten Key entfernen; Rollback bei rotem Test: neuen additiven Key best-effort wieder entfernen; Store-Wechsel erst nach grünem Test) liegt **NICHT** im `VpsProvisioner`, sondern in einer **aufrufenden Schicht** — einem **Rotation-Service/Endpoint** (`POST /api/settings/ssh-keys/{user}/rotate`, **#118**). Diese Schicht:
  - erzeugt das neue ed25519-Keypair ([[ssh-key-generation]]-Mechanik),
  - ruft die `VpsProvisioner`-Primitive in der festgelegten Reihenfolge (`addAuthorizedKey` neu → `testConnection` neu → `removeAuthorizedKey` alt),
  - bezieht den **alten** Public-Key (für das spätere Entfernen) und legt nach grünem Test den **neuen** Key (Private + Public) als aktiven Store-Eintrag ab (`CredentialStore`, ADR-007),
  - hält die Rollback-/Fehler-Semantik (AC3/AC5/AC6 der [[ssh-key-rotation]]) und die HTTP-/`errorClass`-Abbildung (`rotation-verify-failed` → 502 u.a.).
  - Der `VpsProvisioner` bleibt **state-/orchestrierungs-frei**: er kennt keine „alt/neu"-Semantik, keine Reihenfolge, keinen Store-Wechsel — er führt je Aufruf **genau eine** SSH-Operation aus. So bauen #117 (Primitive) und #118 (Orchestrierung) **gegeneinander an den oben fixierten Signaturen**, ohne sich zu blockieren.

- **Provider-Neutralität / Verhältnis zu ADR-009.** Wie #47 ist die Rotation **SSH-from-Backend**, provider-unabhängig, **unabhängig** von der `VpsProviderRegistry` (ADR-009). Kein Provider-API-Call für Key-Rotation.

**Schutz (Floor, prüfbar).** Beide neuen Operationen sind **hoch-privilegiert** (verändern Server-Zugang) und stehen unter demselben Schutz wie #47/ADR-007:
- Hinter der **Access-Mauer**; der Rotation-Endpoint (#118) zusätzlich **identitäts-/rollengeschützt** mit **derselben `CRED_ADMIN_EMAILS`-Logik wie ADR-007** (gesetzt → nur Allowlist mutiert; leer → jede gültige Access-Identität; sonst 403).
- **Audit-First:** Der Verbindungstest **und** das Entfernen des alten Keys werden je auditiert (Identität, Rollen-Label, Ziel `host/targetUser`, Aktion `ssh-key-rotate` mit Phasen-Vermerk, Ergebnis, optional `hostKeyHash`) **ohne** Private-Key-/Geheim-Klartext; schlägt der Audit-Write fehl, unterbleibt die jeweilige Aktion (AC7).
- **Explizites `{result, reason?}` / `{ok, reason?}` ohne Geheim-Leak** (gleiche `sanitizeErrorReason`-Linie wie #47): kein Private-Key (alt/neu), keine Passphrase, kein interner Pfad im `reason`. **Nie** ein Private-Key in Response/Log/Audit/WS (AC8). Der einzige Pfad, über den Private-Key-Klartext das Backend verlässt, bleibt der auditierte Export aus [[ssh-key-generation]] — **nicht** die Rotation.
- **Fail-closed bei `removeAuthorizedKey`:** kann die Datei nicht zuverlässig gelesen/gefiltert/atomar geschrieben werden → `result: "error"` (kein Teil-Schreiben, kein Beschädigen anderer Keys); die Orchestrierung wertet das nach grünem Test als „neuer Key bereits aktiv, alter nicht entfernt" (`result: "rotated", oldKeyRemoved: false` + Warn-`reason`, kein Lockout — AC-Edge-Case der [[ssh-key-rotation]]).

**Verhältnis zu ADR-005/006/007.** ADR-005 gewahrt: kein neuer Store — die einzige Persistenz ist der Key im ADR-007-`CredentialStore` (Betreiber-Credential). ADR-006 gewahrt: **keine** neue Laufzeit-Abhängigkeit — beide Primitive nutzen die bereits vorhandene `ssh2`-Linie (#47); Key-Generierung über Node-Builtins ([[ssh-key-generation]]). ADR-007 gewahrt: Private-Key-Klartext verlässt den Store nur store-intern an den `VpsProvisioner` (für den Test) bzw. an die Orchestrierung (für den Store-Wechsel), **nie** Richtung HTTP/Log/Audit.

**Konsequenzen (prüfbar — Review-Kriterien).**
- **#117:** `VpsProvisioner` exponiert `removeAuthorizedKey({ …, publicKey, … })` + `testConnection({ …, privateKey, … })` mit den oben fixierten Signaturen/Rückgaben. *(architecture/R08-1)*
- **#117:** `removeAuthorizedKey` matcht über den normalisierten Key-Blob `(<type>, <base64>)`, **nicht** über Kommentar/ganze Zeile; entfernt **alle** Blob-Duplikate, lässt jede andere Zeile bytegenau stehen (testbar: `authorized_keys` mit altem+neuem+fremdem Key → nur der adressierte Blob verschwindet, fremde Keys bleiben byteidentisch). *(architecture/R08-2)*
- **#117:** `removeAuthorizedKey` schreibt `authorized_keys` **atomar** (Tempfile + `chmod 600` + `mv`), kein In-place-`sed`; idempotent (`already-absent`, kein Fehler, kein Duplikat). *(architecture/R08-3)*
- **#117:** `testConnection` liefert `ok: true` **nur** bei erfolgreichem Auth-Handshake **und** `true`-exit-0; jeder andere Ausgang → `ok: false` (kein false-positive — testbar mit falschem Key → `ok: false`). *(architecture/R08-4)*
- **#117:** Beide Primitive sind der **einzige** SSH-Weg (kein neuer SSH-Client außerhalb `VpsProvisioner`; Grep: `ssh2`-`Client` nur in `src/VpsProvisioner.js`); identischer Host-Key-/Timeout-/Fehler-Pfad wie #47. *(ADR-008/R01)*
- **#118:** Die Rotation-Orchestrierung lebt **außerhalb** des `VpsProvisioner` (Rotation-Service/Endpoint); der `VpsProvisioner` enthält **keine** gen→add→test→remove-Reihenfolge, keinen Rollback, keinen Store-Wechsel (Grep: kein Keygen/kein `CredentialStore.put` im `VpsProvisioner`). *(architecture/R08-5)*
- **#118:** Der alte Public-Key wird **ausschließlich nach grünem `testConnection`** via `removeAuthorizedKey` entfernt (testbar: roter Test → kein `removeAuthorizedKey`-Aufruf des alten Keys, additiver neuer Key best-effort zurückgerollt). *(ssh-key-rotation AC3/AC5)*
- Verbindungstest **und** Key-Remove je Audit-First; kein Private-Key/Geheimnis in Audit/Log/Response/WS (testbar: Audit/Response enthalten keinen Private-Key — AC7/AC8).
- **Neue Env-Vars: keine** über ADR-007 hinaus (`CRED_MASTER_KEY`, optional `CRED_ADMIN_EMAILS` werden wiederverwendet).

### ADR-009 (Vollfassung) · Multi-Provider-VPS-Boundary (`VpsProviderRegistry` + REST-Adapter)

**Status:** ENTSCHIEDEN · 2026-06-08 · Entscheider: `architekt` · betrifft [[vps-provider-boundary]] (AC1–AC10), [[vps-cloud-init-setup]], [[vps-ssh-key-assignment]], [[view-vps]] v2. Vertagt: [[vps-rebuild-backup]] (Platzhalter).

**Kontext.** Die VPS-Ansicht soll Server bei **drei** Cloud-Providern (Hetzner, IONOS, Hostinger) verwalten: auflisten, starten, stoppen, neu erstellen (Create-from-scratch mit cloud-init-Default-Setup). Jeder Provider hat eigene Auth, eigene Produktlinien und eine eigene Lifecycle-Semantik; nicht jeder unterstützt jede Aktion programmatisch. Höhere Schichten ([[view-vps]]) sollen **keinen** Provider-Code kennen, sondern nur ein provider-agnostisches `VpsMachine`-Read-Model + Capability-Flags. Provider-Tokens liegen verschlüsselt im `CredentialStore` (ADR-007, Schlüssel `credentials/vps/<provider>_api_token`) und dürfen das Backend nie verlassen. Read-Status bleibt live (ADR-005-Linie, kein Maschinen-Store). Rahmen: **ein Dienst, SDK-frei** (ADR-006), Node-Vollstack mit eingebautem `fetch` (Node 22+, `js/R05`).

**Betrachtete Optionen.**

*Option-Achse 1 — Provider-Anbindung (SDK vs. direkter REST):*
1. **Direkter REST-Call über das eingebaute `fetch` je Provider, kein Provider-SDK.** *(gewählt)* Alle drei Provider-APIs sind klassische HTTPS-/JSON-REST-APIs mit Bearer-Token-Auth, die ohne SDK ansprechbar sind: **Hetzner Cloud API** (`https://api.hetzner.cloud/v1`, Bearer-Token, ausgereift, vollständiger `servers`-/`actions`-Lifecycle inkl. `user_data` beim Create) — analog zu IONOS und Hostinger über REST gut bedienbar; **IONOS** — bewusste Wahl der **IONOS-Cloud-(Compute-Engine-)API** (`https://api.ionos.com/cloudapi/v6`, Token-Auth, voller programmatischer Lifecycle inkl. cloud-init-`userData` base64) statt der reinen „VPS"-Produktlinie, da Letztere keinen vergleichbar vollständigen Programmatik-Lifecycle bietet; **Hostinger** — die neuere **Hostinger VPS API** (`https://developers.hostinger.com`, Bearer-Token) deckt list/start/stop/create ab, ist aber dünner/jünger als Hetzner. Diese Wahl hält ADR-006 („SDK-frei") strikt ein, vermeidet drei zusätzliche, unterschiedlich gepflegte Abhängigkeiten samt transitivem Footprint und passt exakt zum bereits etablierten Muster der `GitHubReader`/`GitHubWriter`-Boundaries (REST via `fetch`, kein Octokit-SDK). Wartbarkeit wird nicht über ein SDK, sondern über **einen schmalen Adapter pro Provider** mit normalisiertem Vertrag und zentral getesteten Request-/Response-Mappings gewonnen.
2. **Offizielle/Community-SDKs je Provider** (z.B. `hcloud`-JS für Hetzner, `@ionos-cloud/sdk-nodejs`, ein Hostinger-Client). *Verworfen:* widerspricht ADR-006 direkt; drei zusätzliche Abhängigkeiten mit je eigener Release-Kadenz, Auth-Abstraktion und Update-Risiko; uneinheitliche Reife (Hetzner gut, Hostinger-SDK jung/dünn) → die Abstraktion müsste ohnehin wieder vereinheitlicht werden. Der Mehrwert (getippte Endpunkte) wiegt den Dependency-/Wartungs-Footprint für vier Lifecycle-Operationen nicht auf. Bleibt als spätere, **adapter-lokal gekapselte** Migration offen, falls ein Provider seine REST-API destabilisiert.
3. **Gemischt (SDK für Hetzner, REST für IONOS/Hostinger).** *Verworfen:* zwei Aufruf-Paradigmen im selben Boundary erschweren das gemeinsame Timeout-/Fehler-/Token-Handling und die Tests; Uniformität ist hier mehr wert als der punktuelle SDK-Komfort.

*Option-Achse 2 — cloud-init-Eigentümerschaft:*
4. **Eigene, vom Provider-Boundary getrennte Komponente `CloudInitBuilder`** (`src/vps/CloudInitBuilder.js`) besitzt die **eine** versionierte user-data-Vorlage. *(gewählt)* [[vps-cloud-init-setup]] verlangt explizit **ein zentralisiertes, versioniertes** Default-Setup („nicht über Provider-Adapter verstreut", NFR Wartbarkeit + AC8). Der `CloudInitBuilder` erzeugt aus `{ name, sshPublicKeys: { root, alex } }` einen validierten `#cloud-config`-String; die Provider-Adapter sind reine **Transporteure** der fertigen user-data an ihre jeweilige Create-API (Hetzner `user_data`, IONOS base64-`userData`, Hostinger user-data-Feld). So bleibt die Setup-Logik provider-unabhängig und an genau einer Stelle testbar.
5. **cloud-init je Adapter selbst bauen.** *Verworfen:* verstößt gegen die Zentralisierungs-Forderung, vervielfacht die Vorlage und macht AC1–AC8 von [[vps-cloud-init-setup]] pro Provider erneut prüfbedürftig.

**Entscheidung (bindend).**

- **Boundary & Layout.** Genau **eine** Boundary `VpsProviderRegistry` (`src/vps/VpsProviderRegistry.js`) ist der einzige Ort, der Provider-APIs anspricht (AC1, analog R01 zu `CredentialStore`). Sie löst je Provider-Id (`hetzner` | `ionos` | `hostinger`) einen Adapter auf:
  ```
  src/vps/
    VpsProviderRegistry.js     # Auflösung, Token-Bezug, Aggregation, Degradation
    CloudInitBuilder.js        # die eine versionierte cloud-init-Vorlage (vps-cloud-init-setup)
    providers/
      hetzner.js               # REST-Adapter (api.hetzner.cloud/v1)
      ionos.js                 # REST-Adapter (api.ionos.com/cloudapi/v6 — Cloud/Compute-Engine)
      hostinger.js             # REST-Adapter (Hostinger VPS API)
    normalize.js               # Provider-Rohantwort → VpsMachine (gemeinsame Mapping-Helfer)
  ```
  Die HTTP-Route-Handler (`/api/vps/*`) rufen **ausschließlich** die Registry, nie einen Adapter direkt; kein anderes Modul importiert `providers/*` oder spricht eine Provider-URL an (Grep-prüfbar).

- **`VpsProvider`-Vertrag (gemeinsame Signatur, je Adapter identisch).**
  - `capabilities()` → `{ list, start, stop, create }` (boolean je Aktion, provider-statisch deklariert).
  - `listMachines()` → `VpsMachine[]` (normalisiert).
  - `start(serverId)` / `stop(serverId)` → `{ result: "ok"|"unsupported"|"error", reason? }`.
  - `create({ name, region, serverType, image, userData, sshPublicKeys })` → `VpsMachine`.
  Ein Adapter, dessen Provider eine Lifecycle-Aktion API-seitig nicht kennt, deklariert das Flag in `capabilities()` als `false` **und** liefert bei direktem Aufruf `result: "unsupported"` (HTTP 422) — er **simuliert nichts** und löst keine destruktive Ersatzaktion aus (AC6, [[vps-provider-boundary]] AC6).

- **Normalisiertes Maschinen-Modell `VpsMachine`.** `{ provider, serverId, name, status, ipv4, ipv6, region, serverType, createdAt }`. `status` wird je Adapter auf das gemeinsame Enum `running | stopped | provisioning | error | unknown` gemappt (`normalize.js`). **Fehlende/unbekannte Felder werden zu `null`** (bzw. `status: "unknown"`), **nie** zu einem Fehler — ein Provider, der z.B. keine IPv6 oder kein `createdAt` liefert, kippt weder das Mapping noch die Aggregation. Das vollständige Schema steht in [[data-model]] (Abschnitt `VpsMachine`) und ist dort die kanonische Referenz für `coder`/`tester`.

- **Capability-Flag-Mechanik.** `GET /api/vps/providers` liefert `[{ id, configured, capabilities }]`. `configured` = Token im `CredentialStore` gesetzt; ein nicht konfigurierter Provider liefert `configured: false` und löst **keinen** Provider-API-Aufruf aus (AC2). `capabilities` stammt aus dem statischen `capabilities()` des Adapters und steuert die UI ([[view-vps]] AC6: nicht unterstützte Aktion disabled).

- **Token-Injektion ohne Persistenz.** Pro Aufruf zieht die **Registry** (nicht der Adapter) das Token store-intern über `CredentialStore.get('credentials/vps/<provider>_api_token')` und reicht es **als Funktions-Argument** transient an den Adapter (Adapter setzen den `Authorization: Bearer …`-Header lokal in der `fetch`-Option). Das Token wird **nie** in ein Modul-Feld/Closure über den Request hinaus gecacht, **nie** in eine URL/Argv/Query gesetzt (immer Header), **nie** geloggt/auditiert/in eine Response gespiegelt. Fehlt das Token → `provider-not-configured` (kein Provider-Call); ungültiges Token → `provider-auth-failed` (502) ohne Token-Leak (Edge-Cases der Spec). Das ist exakt das transiente Mint-/Inject-Muster der `GitHubWriter`-Boundary, übertragen auf statische Provider-Tokens.

- **cloud-init-Erzeugung (Owner = `CloudInitBuilder`).** Der Create-Pfad in der Registry (1) löst über [[vps-ssh-key-assignment]] die Label→Public-Key-Zuordnung `{ root, alex }` store-intern auf (nur **Public**-Keys, ADR-007-Metadatum; nie Private-Keys), (2) ruft `CloudInitBuilder.build({ name, sshPublicKeys: { root, alex } })` → validierter `#cloud-config`-String, (3) bricht **vor** dem Provider-Call mit 422 `missing-ssh-key` ab, wenn ein Public-Key für root oder alex fehlt ([[vps-cloud-init-setup]] AC7, [[vps-ssh-key-assignment]] AC5), (4) übergibt die fertige user-data dem adressierten Adapter, der sie provider-korrekt einbettet (IONOS base64-kodiert). Die Vorlage ist versioniert (AC8) und liegt **nur** im `CloudInitBuilder`, nicht in den Adaptern.

- **Read-Aggregation live + Degradation (ADR-005-Linie).** `GET /api/vps/machines` ruft `listMachines()` über **alle konfigurierten** Provider auf (kein Maschinen-Store). Die Registry führt die Provider-Calls mit **Per-Provider-Timeout** (`fetch` + `AbortController`, Vorschlag ~10 s, `coder` finalisiert) und sammelt das Ergebnis **degradierend**: erfolgreiche Provider liefern ihre `VpsMachine[]`, ein fehlerhafter/unerreichbarer Provider erzeugt **keinen** 500 der Gesamt-Antwort, sondern einen Eintrag in `providerErrors: [{ provider, errorClass }]` (AC4, analog `factory-status`-Degradation). Mutierende Einzel-Aktionen (start/stop/create) degradieren **nicht** — sie melden ihren Fehler klar als `{ result: "error", reason }` ohne Teilzustand.

- **Fehler-/`unsupported`-Semantik (kanonisch).** `errorClass`-Werte: `provider-not-configured` (422, kein Call), `provider-auth-failed` (502), `not-found` (404, Adapter-durchgereicht), `provider-unavailable` (502/503, Rate-Limit/Timeout). `unsupported` ist **kein** Fehler-, sondern ein Capability-Ergebnis (422, `result: "unsupported"`). Idempotenz: start auf laufendem / stop auf gestopptem Server → `result: "ok"` (Adapter mappt den bereits-im-Zielzustand-Fall, kein Fehler).

**Schutz (Floor, prüfbar).**
- Alle `/api/vps/*`-Endpunkte hinter der **Access-Mauer** (403 ohne gültigen Access, AC9).
- Mutierende Aktionen (start/stop/create) zusätzlich **identitäts-/rollengeschützt** mit **derselben `CRED_ADMIN_EMAILS`-Logik wie ADR-007** (gesetzt → nur Allowlist mutiert, sonst 403; leer → jede gültige Access-Identität). **Kein** zusätzlicher Bestätigungstoken in diesem Durchgang (Create/Start/Stop sind nicht destruktiv).
- **Audit-First:** vor jeder Mutation ein `AuditEntry` (Identität, Provider, serverId/Name, Aktion, Zeit) **ohne** Token; schlägt der Audit-Write fehl → Aktion unterbleibt (AC10).
- Provider-Tokens at rest verschlüsselt (ADR-007) und **nie** in Response/Log/Audit/WS/Argv/Frontend-Bundle (AC10). cloud-init enthält **ausschließlich Public-Keys**, nie Private-Key-Material oder Provider-Tokens ([[vps-cloud-init-setup]] AC6).

**Vertagt (bewusst, nicht in diesem Durchgang).** **Rebuild** (destruktives Reimage eines bestehenden Servers) und **Backup/Snapshot** bleiben Platzhalter ([[vps-rebuild-backup]]). Wenn sie kommen, docken sie als **weitere Capability-Flags/Methoden** an denselben `VpsProvider`-Vertrag an (kein neuer Boundary). Das destruktive **Rebuild erhält dann eine stärkere Guardrail-Stufe als Create/Start/Stop** — über Access + Audit hinaus einen **expliziten Bestätigungstoken** (type-to-confirm Servername), so dass der hier festgelegte „Access + Audit, kein Zusatztoken"-Schutz nur für die nicht-destruktiven Lifecycle-Aktionen gilt.

**Verhältnis zu ADR-005 / ADR-006.**
- **ADR-005** bleibt gewahrt: VPS-Maschinen sind **Read-Models live aus den Provider-APIs**, kein persistenter Maschinen-State-Store. Die einzige Persistenz im VPS-Umfeld sind die Provider-Tokens — und die liegen im ADR-007-`CredentialStore` (Betreiber-Credential, nicht Fabrik-/Domänen-State; gleiche Re-Scoping-Linie wie ADR-007).
- **ADR-006** bleibt gewahrt: **kein** Provider-SDK, **keine** neue Laufzeit-Abhängigkeit — alle drei Adapter nutzen das in Node eingebaute `fetch` (`js/R05`) mit `AbortController`-Timeout (`js/R03`). Ein Dienst, SDK-frei.

**Konsequenzen (prüfbar — Review-Kriterien).**
- `VpsProviderRegistry` ist der **einzige** Modul, der eine Provider-URL/`fetch` gegen Hetzner/IONOS/Hostinger ausführt (Grep: kein `api.hetzner.cloud` / `api.ionos.com` / Hostinger-Host außerhalb von `src/vps/providers/*`). *(architecture/R01)*
- Jeder der drei Adapter implementiert dieselbe `VpsProvider`-Signatur (`capabilities`/`listMachines`/`start`/`stop`/`create`). *(architecture/R02)*
- Kein Provider-SDK in `package.json`-Dependencies; keine `node-fetch`-Abhängigkeit (eingebautes `fetch`). *(ADR-006, js/R05)*
- Die cloud-init-Vorlage existiert **genau einmal** (`CloudInitBuilder`), kein `#cloud-config` in einem `providers/*`-Adapter.
- Kein Provider-Token in Response/Log/Audit/WS/Argv/URL/Frontend-Bundle (testbar: Response-Body/Audit enthalten den gesetzten Token nicht — AC10).
- Read-Aggregation liefert bei Ausfall eines Providers 200 mit `providerErrors`, kein 500 (AC4).
- Nicht unterstützte Lifecycle-Aktion → `capabilities().<aktion> === false` **und** 422 `result: "unsupported"`, keine Ersatzaktion (AC6).
- Mutationen hinter Access + `CRED_ADMIN_EMAILS`-Rollenschutz + Audit-First (AC9/AC10).
- **Neue Env-Vars: keine** über ADR-007 hinaus (`CRED_MASTER_KEY`, optional `CRED_ADMIN_EMAILS` werden wiederverwendet). Provider-Tokens werden über die bestehende Settings-/Credentials-Capability ([[settings-credentials]] AC9) gesetzt.

### ADR-010 (Vollfassung) · Cloudflare-API-Boundary (`CloudflareApi`) + Tunnel-Verwaltungs-Modell

**Status:** ENTSCHIEDEN · 2026-06-08 · Entscheider: `architekt` · betrifft [[view-cloudflare]] v2 (Capability A), [[deploy-lifecycle]] (Capability B), [[cloudflare-reconciliation]] (Capability C), [[settings-credentials]] (Cloudflare-Sektion). Baut auf ADR-005 (kein State-Store), ADR-006 (ein Dienst, SDK-frei, eingebautes `fetch`), ADR-007 (CredentialStore), ADR-009 (REST-Adapter-/Token-Injektions-Muster).

**Kontext.** Die Cloudflare-Ansicht soll von einem Platzhalter zu einem Inventar- + Lösch-Werkzeug reifen, und der Betreiber will einen panel-steuerbaren, selbst-heilenden Deploy-Mechanismus (heute manuell im `preview`-Skill: ghcr-Image → Container → Cloudflare-Tunnel-Route + DNS-CNAME `<app>.<domain>`, mit Cleanup). Alle drei Capabilities (A/B/C) brauchen **einen** Ort, der die Cloudflare-API anspricht — Zones lesen, Tunnel + Public-Hostname-Routen lesen/mutieren, DNS-Records anlegen/löschen. Die zentrale, prägende Weiche ist das **Tunnel-Verwaltungs-Modell**, weil es bestimmt, ob A/B/C SSH brauchen oder rein API+Docker laufen. Rahmen: ein Dienst, SDK-frei, Node 22+ mit eingebautem `fetch` (`js/R05`), Cloudflare-Token verschlüsselt im `CredentialStore` (ADR-007). Im `preview`-Skill existieren bereits die Fabrik-Env-Vars `CLOUDFLARE_API_TOKEN` / `CLOUDFLARE_ACCOUNT_ID` / `CLOUDFLARE_ZONE_ID`.

**Betrachtete Optionen.**

*Option-Achse 1 — Tunnel-Verwaltungs-Modell (die zentrale Weiche):*
1. **Cloudflare-remote-managed Tunnels** — Ingress-/Public-Hostname-Konfiguration über die Cloudflare-API (`GET/PUT /accounts/{accId}/cfd_tunnel/{tunnelId}/configurations`); `cloudflared` läuft **token-basiert** und zieht seine Config aus Cloudflare. dev-gui verwaltet Routen rein über die API, **kein SSH** für den Route-Teil; Reconciliation = Cloudflare-API (Routen) + Docker-Read pro VPS (Container). *(gewählt)*
2. **cloudflared-config-on-VPS** (heutiges Modell, BOOTSTRAP §5 / `preview`-Skill) — Ingress in lokaler `config.yml`, `cloudflared`-Reload (`systemctl restart cloudflared`). Routen-Mutation + Reconciliation brauchen **SSH** auf jeden VPS (an die `VpsProvisioner`-SSH-Boundary aus ADR-008 angedockt). *Verworfen als Zielmodell* (s. Abwägung).

*Option-Achse 2 — Provider-Anbindung (SDK vs. REST):*
3. **Direkter REST-Call über eingebautes `fetch`, kein Cloudflare-SDK.** *(gewählt)* Die Cloudflare-API ist eine klassische HTTPS-/JSON-REST-API mit Bearer-Token-Auth (`Authorization: Bearer <token>`); Zones/DNS/`cfd_tunnel` sind ohne SDK ansprechbar. Hält ADR-006 strikt ein, passt exakt zum etablierten `GitHubReader`/`VpsProviderRegistry`-Muster.
4. **`cloudflare`-JS-SDK.** *Verworfen:* widerspricht ADR-006 (SDK-frei); zusätzliche Abhängigkeit mit eigener Release-Kadenz für eine schmale Endpunkt-Menge. Bleibt boundary-lokal kapselbare Migration, falls die REST-API destabilisiert.

**Abwägung der zentralen Weiche (Modell i vs. ii) — Entscheidungskriterien.**
- **Migrationskosten vom heutigen Modell:** Modell ii ist der Ist-Zustand (`preview`-Skill editiert `config.yml` + `cloudflared tunnel route dns` + Reload). Modell i verlangt, jeden produktiven Tunnel **einmalig** von „locally-managed" auf „remotely-managed" umzustellen (Tunnel ohne lokale `config.yml`, nur Token; Ingress wandert in die Cloudflare-Config-API). Das ist eine **einmalige, gut umrissene Umstellung pro VPS**, kein laufender Aufwand.
- **Komplexität / Fit zu ADR-005/006:** Modell i hält die **gesamte** Cloudflare-Mutation in **einem** API-Boundary (`CloudflareApi`) — kein zweiter, parallel zu wartender SSH-Mutationspfad für Routen, keine `config.yml`-Parser/Renderer/Reload-Choreografie über mehrere Hosts. Modell ii zwingt jede Routen-Mutation **und** die Reconciliation über SSH (ADR-008-Linie) und macht die Tunnel-Config zu host-lokalem Datei-State, der je VPS abweichen kann — schwerer live abgleichbar (Reibung mit ADR-005). Modell i hält den Read-/Mutate-Status **live aus der Cloudflare-API** (eine Quelle der Wahrheit für Routen), ADR-005-konform.
- **Self-Lockout-Risiko:** in **beiden** Modellen besteht es; in Modell i ist der `LockoutGuard` (ADR-011) **vor genau einem** Mutationspfad (der API) wirksam — eine einzige prüfbare Stelle. In Modell ii müsste der Schutz zusätzlich den `config.yml`-Schreibpfad + DNS-Pfad pro Host abdecken (zwei Wege, größere Fläche).
- **Capability-C-Fit:** Reconciliation braucht den **Routen-Bestand** als Vergleichsmenge. In Modell i kommt er aus **einem** API-Call (`GET …/configurations` je Tunnel) statt aus dem Parsen von `config.yml` über N SSH-Verbindungen.

**Entscheidung (bindend).**
- **Tunnel-Modell = (i) remote-managed.** dev-gui verwaltet Tunnel-Ingress/Public-Hostname-Routen **ausschließlich über die Cloudflare-API**. `cloudflared` läuft auf den VPS token-basiert (remotely-managed) und zieht seine Config aus Cloudflare. **Für den Cloudflare-Route-Teil von A/B/C ist kein SSH nötig** (SSH bleibt nur für den **Docker-Container**-Teil von B/C relevant, s. ADR-012).
- **Boundary & Layout.** Genau **eine** Boundary `CloudflareApi` (`src/cloudflare/CloudflareApi.js`) ist der einzige Ort, der eine Cloudflare-API-URL via `fetch` anspricht (analog R01 zu `CredentialStore`/`VpsProviderRegistry`). Layout:
  ```
  src/cloudflare/
    CloudflareApi.js     # einziger fetch-Ort: zones, dns_records, cfd_tunnel(+configurations)
    LockoutGuard.js      # Self-Lockout-Prädikat (ADR-011) — netz-/store-los
    normalize.js         # Cloudflare-Rohantwort → CfZone / CfTunnel / CfRoute (Mapping-Helfer)
  ```
  Die HTTP-Route-Handler (`/api/cloudflare/*`) rufen **ausschließlich** `CloudflareApi`; kein anderes Modul spricht eine Cloudflare-URL an (Grep-prüfbar). `DeployOrchestrator` (ADR-012) und `ReconciliationJob` (ADR-013) konsumieren `CloudflareApi` als Bibliotheks-Boundary, nicht die HTTP-Endpunkte.
- **Read-/Mutate-Trennung im selben Boundary.** `CloudflareApi` exponiert getrennte Methoden-Gruppen: **read** (`listZones()`, `listTunnels(zoneId)`, `listRoutes(tunnelId)`) ohne Seiteneffekt; **mutate** (`putTunnelConfig(tunnelId, ingress)`, `createDnsRecord(...)`, `deleteDnsRecord(...)`, `deleteTunnel(tunnelId)`, `removeRoute(tunnelId, hostname)`). Jede `mutate*`-Methode fragt **zuerst** `LockoutGuard.isProtected(target)` (ADR-011) und führt bei `true` **gar keinen** API-Call aus.
- **Normalisiertes Read-Modell.** `CfZone { id, name, status }`, `CfTunnel { id, name, status, zoneId }`, `CfRoute { hostname, service, tunnelId, protected: boolean }`. Fehlende Felder → `null`, **nie** Fehler (ADR-009-Linie). `protected` wird beim Mappen über `LockoutGuard` gesetzt, damit das Frontend die Lösch-Affordance unterdrücken kann.
- **Token-/Account-Injektion ohne Persistenz.** Pro Aufruf zieht `CloudflareApi` Token + Account-Id store-intern über `CredentialStore.get('credentials/cloudflare/api_token')` bzw. `…/account_id` und setzt das Token **immer als Header** (`Authorization: Bearer …`), **nie** in URL/Query/Argv; **nie** geloggt/auditiert/in eine Response gespiegelt; **nie** über den Request hinaus gecacht. Exakt das transiente Inject-Muster aus ADR-009. **Zone-Id wird live aufgelöst** (`listZones()`), **nicht** als Credential gepflegt — Domänen-Anwahl (Capability A) bedeutet „Zone aus der Live-Liste wählen".
- **Read-Aggregation live + Degradation (ADR-005-Linie).** Zonen/Tunnel/Routen werden **live** gelesen (kein Tunnel-Store). Mehr-Zonen-/Mehr-Tunnel-Lesungen laufen mit Per-Call-Timeout (`fetch` + `AbortController`, ~10 s, `coder` finalisiert) und degradieren pro Zone: ein fehlerhafter Zone-/Tunnel-Call kippt nicht die Gesamt-Antwort, sondern erzeugt `errors: [{ scope, errorClass }]` (analog `providerErrors` in ADR-009).
- **Fehler-Semantik (kanonisch).** `cloudflare-not-configured` (422, kein Call — Token/Account fehlt), `cloudflare-auth-failed` (502, ungültiger Token, ohne Leak), `not-found` (404 durchgereicht), `cloudflare-unavailable` (502/503, Rate-Limit/Timeout), `protected-resource` (422 — LockoutGuard, ADR-011).

**Verhältnis der Fabrik-Env-Vars (`CLOUDFLARE_*`) zum `CredentialStore`.** Die im `preview`-Skill genutzten `CLOUDFLARE_API_TOKEN` / `CLOUDFLARE_ACCOUNT_ID` / `CLOUDFLARE_ZONE_ID` bleiben **Eigentum der Fabrik-Skills** (im Plugin-Repo, via `.env.gpg`) und werden **nicht** migriert — sie bedienen die manuelle `preview`-Mechanik. dev-gui führt seine **eigenen** Cloudflare-Credentials im `CredentialStore` (`credentials/cloudflare/api_token`, `…/account_id`) — **Koexistenz, keine Migration**. *Begründung:* die beiden Pfade haben unterschiedliche Lebenszyklen (Plugin-Env beim Deploy vs. GUI-gesetztes Credential zur Laufzeit) und unterschiedliche Geheimnis-Trennung (ADR-007 write-only); sie zu koppeln brächte keinen Mehrwert und verletzte die write-only-Linie. Beide dürfen denselben physischen Cloudflare-Token-Wert verwenden — das ist Betreiber-Wahl, kein Architektur-Zwang.

**Schutz (Floor, prüfbar).**
- Alle `/api/cloudflare/*`-Endpunkte hinter der **Access-Mauer** (403 ohne gültigen Access).
- Mutierende Aktionen (Routen-/Tunnel-/DNS-Löschen, Route-Anlegen) zusätzlich **identitäts-/rollengeschützt** mit **derselben `CRED_ADMIN_EMAILS`-Logik wie ADR-007** (gesetzt → nur Allowlist; leer → jede gültige Access-Identität).
- **Audit-First:** vor jeder Mutation ein `AuditEntry` (Identität, Zone/Tunnel/Hostname, Aktion, Zeit) **ohne** Token; schlägt der Audit-Write fehl → Aktion unterbleibt.
- **Self-Lockout-Hard-Block** (ADR-011): protected Ziel → 422, keine Mutation, vor Audit/Confirm.
- **type-to-confirm** für jede destruktive Tunnel-/Routen-Löschung (ADR-011).
- Cloudflare-Token + Account-Id **nie** in Response/Log/Audit/WS/Argv/URL/Frontend-Bundle. `cloudflared`-/Tunnel-Tokens werden von dev-gui **nicht** verwaltet/geleakt (remote-managed cloudflared zieht seinen eigenen Token außerhalb von dev-gui).

**Migrationspfad vom heutigen Modell (benannt, nicht in diesem Durchgang erzwungen).** Heutige VPS-Tunnel sind locally-managed (`config.yml`, BOOTSTRAP §5). Für Modell i: pro VPS den produktiven Tunnel auf remotely-managed umstellen (Cloudflare-Dashboard/API: Tunnel-Config in die `configurations`-API überführen, lokale `config.yml`-Ingress entfernen, `cloudflared` mit reinem Token starten). **Die `devgui`-Route selbst** muss vor der Umstellung als protected im `LockoutGuard` erkannt werden, damit sie nie versehentlich von dev-gui mutiert wird. **O1 — ENTSCHIEDEN 2026-06-08 (Betreiber): bewusst vertagt.** Ob/wann die Bestands-Tunnel migriert werden, ist eine separate Betriebsentscheidung (nicht in diesem Durchgang erzwungen); A/B/C werden gegen das remote-managed-Zielmodell gebaut, bis zur operativen Migration eines VPS bleibt dessen `config.yml`-Tunnel unberührt. Reihenfolge bei späterer Migration: `devgui`-Route zuerst als protected im `LockoutGuard` verankern (via `DEVGUI_HOSTNAME`, O2), dann VPS für VPS umstellen.

**Verhältnis zu ADR-005 / ADR-006.**
- **ADR-005** gewahrt: Cloudflare-Inventar (Zones/Tunnel/Routen) ist **Read-Model live aus der Cloudflare-API**, kein persistenter Store. Einzige Persistenz: der Cloudflare-Token im ADR-007-`CredentialStore` (Betreiber-Credential).
- **ADR-006** gewahrt: **kein** Cloudflare-SDK, **keine** neue Laufzeit-Abhängigkeit — `fetch` (`js/R05`) + `AbortController`-Timeout (`js/R03`).

**Konsequenzen (prüfbar — Review-Kriterien).**
- `CloudflareApi` ist der **einzige** Modul, der eine Cloudflare-API-URL/`fetch` ausführt (Grep: kein `api.cloudflare.com` / `cfd_tunnel` / `dns_records` außerhalb von `src/cloudflare/`). *(architecture/R01)*
- Read- und Mutate-Methoden sind getrennt; jede `mutate*`-Methode ruft `LockoutGuard.isProtected` **vor** dem API-Call. *(architecture/R02)*
- Kein Cloudflare-SDK in `package.json`; keine `node-fetch`-Abhängigkeit (eingebautes `fetch`). *(ADR-006, js/R05)*
- Kein Cloudflare-Token/Account-Id in Response/Log/Audit/WS/Argv/URL/Frontend-Bundle (testbar: Response-Body/Audit enthalten den gesetzten Token nicht).
- Read-Aggregation liefert bei Ausfall einer Zone 200 mit `errors`, kein 500.
- Die Fabrik-`CLOUDFLARE_*`-Env-Vars werden **nicht** im dev-gui-Backend gelesen (Grep: kein `process.env.CLOUDFLARE_*` außerhalb evtl. Doku); dev-gui nutzt ausschließlich `credentials/cloudflare/*` aus dem `CredentialStore`.
- Mutationen hinter Access + `CRED_ADMIN_EMAILS`-Rollenschutz + Audit-First.
- **Neue Env-Vars: keine** über ADR-007 hinaus. Cloudflare-Token/Account-Id werden über [[settings-credentials]] gesetzt (`integration=cloudflare`, `name ∈ {api_token, account_id}`).

### ADR-011 (Vollfassung) · Self-Lockout-Schutz (`LockoutGuard` Hard-Block + type-to-confirm)

**Status:** ENTSCHIEDEN · 2026-06-08 · Entscheider: Betreiber (bindend vorgegeben) · architektonisch ausgearbeitet vom `architekt` · betrifft [[view-cloudflare]] v2, [[deploy-lifecycle]], [[cloudflare-reconciliation]]. Baut auf ADR-004 (Access als Mauer), ADR-009 (type-to-confirm-Guardrail-Stufe), ADR-010 (CloudflareApi).

**Kontext.** dev-gui läuft selbst hinter einem Cloudflare-Tunnel (`devgui.<domain>`) + Cloudflare Access (ADR-004). Eine Cloudflare-Mutation, die die `devgui`-Route ODER die Access-Mauer entfernt/verbiegt, würde die Admin-Konsole **selbst aussperren** (Self-Lockout) — irreversibel ohne externen Eingriff. Der Betreiber hat dafür eine bindende Schutz-Entscheidung vorgegeben (nicht neu aufzurollen): Hard-Block + type-to-confirm. Diese ADR setzt sie prüfbar um.

**Betrachtete Optionen** (Umsetzungsform der vorgegebenen Entscheidung).
1. **Eigene reine Prädikat-Komponente `LockoutGuard`, von jedem Mutationspfad vor der Aktion aufgerufen.** *(gewählt)* Netz-/store-los, deterministisch, einzeln testbar; genau **eine** Stelle, die „protected" definiert.
2. Protected-Check verstreut in jedem Handler (`if hostname === devgui …`). *Verworfen:* dupliziert die Erkennungs-Logik über A/B/C, driftet, schwer prüfbar.
3. Reine UI-Sperre (Button disabled). *Verworfen:* UI-only ist kein Schutz — eine direkte API-Anfrage (oder die Reconciliation/Deploy-Boundary) umginge sie. Der Block muss **im Backend, im Mutationspfad** sitzen.

**Entscheidung (bindend).**
- **Komponente.** `LockoutGuard` (`src/cloudflare/LockoutGuard.js`) — eine reine Funktion/Klasse ohne Netz/IO: `isProtected(target) → boolean`, wobei `target` ein Hostname, ein DNS-Record oder eine Tunnel-Route/-Id ist. **Jeder** mutierende Cloudflare-Pfad (`CloudflareApi.mutate*`, ADR-010) fragt `isProtected` **vor** der Aktion; `true` → **422 `protected-resource`**, keine Mutation, **vor** Audit/Confirm/Identitätsprüfung (harte erste Schranke). Gilt **egal** welche Identität, **egal** welcher Confirm-Token — nicht überschreibbar.
- **Erkennungsregeln (kanonisch, allowlist-/musterbasiert).** Protected ist ein Ziel, das (a) dem **eigenen `devgui`-Hostname** entspricht (aus der Deployment-Konfig ableitbar, s.u.), ODER (b) zur **Cloudflare-Access-Mauer** gehört (Access-Application-Hostname / die `devgui`-Route, die Access erzwingt). Erkennung über eine **konfigurierbare Allowlist geschützter Hostnamen/Suffixe** + den live ermittelten `devgui`-Hostname. **Fail-closed bei Mehrdeutigkeit:** kann der Guard nicht sicher entscheiden (z.B. Hostname nicht eindeutig zuordenbar, Konfig fehlt), behandelt er das Ziel **als protected** (lieber eine legitime Löschung blockieren als sich aussperren).
- **`devgui`-Hostname-Quelle (O2 — ENTSCHIEDEN 2026-06-08, Betreiber).** Der eigene Hostname wird aus der **expliziten Env `DEVGUI_HOSTNAME`** bezogen — **nicht** geraten. Diese Env ist die kanonische Quelle für die `devgui`-Erkennung in (a); sie ist in Produktion zu setzen (Doku/Compose). Solange sie nicht gesetzt ist, greift fail-closed für den Bereich „könnte die eigene Route sein" (breitere Sperre, kein Self-Lockout). `coder`/Deployment: `DEVGUI_HOSTNAME` in `docker-compose.yml` als `environment`-Eintrag ergänzen (analog `ACCESS_*`).
- **type-to-confirm (zweite Schranke, für nicht-protected destruktive Aktionen).** Jede **destruktive** Cloudflare-Mutation (Tunnel löschen, Route entfernen, DNS-Record löschen) auf ein **nicht-protected** Ziel erfordert einen **Bestätigungstoken** im Request-Body (`confirm: "<hostname>"`), der **exakt** dem Ziel-Hostname entsprechen muss — analog der Rebuild-Guardrail in ADR-009. Fehlt/mismatcht der Token → **422 `confirmation-required`**, keine Mutation. Reihenfolge: `isProtected` (hart) → `confirm`-Match → `CRED_ADMIN_EMAILS`-Rolle → Audit-First → Mutation.
- **Reconciliation-Ausnahme (begründet).** Der `ReconciliationJob` (ADR-013) löscht verwaiste Routen **und legt fehlende Routen managed Container an** (beidseitige Heilung) **ohne** type-to-confirm (kein interaktiver Akteur), **aber** unter demselben `LockoutGuard`-Hard-Block (protected → **nie löschen, nie anlegen**) + Audit-First. Da die Route-Heilung den **atomaren Anlege-Pfad des `DeployOrchestrator`** (ADR-012) wiederverwendet, durchläuft sie denselben `LockoutGuard`-Check wie ein interaktiver Deploy. Begründung: type-to-confirm schützt vor menschlichem Fehlgriff; der Cron hat keinen Menschen — der maßgebliche Self-Lockout-Schutz (LockoutGuard) gilt in **beide** Heilungs-Richtungen unverändert hart.

**Schutz (Floor, prüfbar).**
- LockoutGuard ist die **einzige** Definition von „protected"; kein Handler dupliziert die Logik (Grep). *(architecture/R02)*
- Block greift **im Backend-Mutationspfad**, nicht nur im UI.

**Verhältnis zu ADR-005/006.** Reine Logik-Komponente, kein Store, keine Abhängigkeit — neutral zu ADR-005/006.

**Konsequenzen (prüfbar — Review-Kriterien).**
- Mutation auf den `devgui`-Hostname / Access-Mauer-Record → **422 `protected-resource`**, kein Cloudflare-API-Call ausgeführt (testbar: API-Mock erhält keinen Delete). *(architecture/R01)*
- Block ist **nicht** durch Identität, `CRED_ADMIN_EMAILS`-Mitgliedschaft oder Confirm-Token überschreibbar (testbar: auch Admin + korrekter Confirm → 422 bei protected).
- Destruktive Aktion auf nicht-protected Ziel **ohne** korrekten `confirm`-Token → **422 `confirmation-required`**, keine Mutation.
- Fehlt der `devgui`-Hostname-Konfig-Wert / ist das Ziel mehrdeutig → fail-closed (als protected behandelt).
- Reconciliation-Lösch- **und** -Anlege-Pfad rufen `LockoutGuard.isProtected` ebenso (Grep: ReconciliationJob nutzt denselben Guard bzw. den ADR-012-Anlege-Pfad, der ihn enthält) — protected wird **weder** gelöscht **noch** angelegt.

### ADR-012 (Vollfassung) · Deploy-Lifecycle-Boundary (`DeployOrchestrator` + `VpsDockerControl`)

**Status:** ENTSCHIEDEN · 2026-06-08 · Entscheider: `architekt` · betrifft [[deploy-lifecycle]] (Capability B), [[cloudflare-reconciliation]] (Capability C, konsumiert `VpsDockerControl`). Baut auf ADR-002 (always-on), ADR-005 (kein State-Store), ADR-008 (SSH-from-Backend-Linie, `VpsProvisioner`), ADR-009 (VPS-Provider-Boundary — **nur Server-Lifecycle**), ADR-010 (CloudflareApi), ADR-011 (LockoutGuard). **Ändert ADR-008/009 nicht** — referenziert/erweitert nur.

**Kontext.** Betreiber-Vision: jedes Mal, wenn ein Image deployed wird, soll es aus GitHub (ghcr) auf einen VPS in einen Docker gepullt und die zugehörige Tunnel-Route zur Domäne erstellt werden; wird ein Container removed, soll auch die Tunnel-Route entfernt werden. Das ist die produktive, panel-steuerbare Variante des heutigen `preview`-Skills. **Wichtige Lücke:** Der VPS-Provider-Boundary (ADR-009) deckt nur **Server-Lifecycle** (list/start/stop/create), **nicht** Container-Deployment auf einem laufenden Server. Cloudflare-Routen kommen rein über die API (ADR-010, kein SSH). Container-Deployment **auf** einem laufenden VPS braucht aber einen Mechanismus, der über ADR-009 hinausgeht.

**Betrachtete Optionen.**

*Option-Achse 1 — Wohnort der Deploy-Orchestrierung:*
1. **Eigener `DeployOrchestrator`-Boundary + eigene Deployments-View/Route.** *(gewählt — folgt der Betreiber-/Orchestrator-Empfehlung)* Cloudflare-View bleibt Inventar+Lösch-Werkzeug (A); Deploy-Orchestrierung ist eine eigene Komponente mit eigener UI.
2. Deploy-Logik in die Cloudflare-View drängen. *Verworfen:* vermischt Inventar (A, read+delete auf Cloudflare) mit einer zwei-System-Orchestrierung (Docker **und** Cloudflare); überlädt die View und den Cloudflare-Boundary mit Docker-Belangen.

*Option-Achse 2 — Container-Zugriff auf den laufenden VPS:*
3. **Neue `VpsDockerControl`-Boundary: `docker pull/run/rm` über die `VpsProvisioner`-SSH-Linie (ADR-008).** *(gewählt)* ADR-008 hat SSH-from-Backend bereits als Mechanik fixiert; Docker-on-VPS ist derselbe SSH-Weg mit `docker`-Kommandos. Eine **neue** Boundary (nicht ADR-008/009 ändern), die ADR-008s `VpsProvisioner`/Private-Key-aus-CredentialStore wiederverwendet.
4. Remote-Docker-API (TCP/TLS, `DOCKER_HOST` über das Netz). *Verworfen:* exponiert die Docker-Engine jedes VPS übers Netz (große Angriffsfläche), braucht TLS-Client-Certs je Host und eine offene Daemon-Port-Policy — deutlich höhere Sicherheits- und Betriebslast als SSH, das es ohnehin schon gibt (ADR-008).
5. Deploy in den Server-Create-Pfad (ADR-009) integrieren. *Verworfen:* ADR-009 ist bewusst auf Server-Lifecycle begrenzt; Deploy auf **bestehende, laufende** Server (Mehrfach-Deploy pro VPS) passt nicht in den Create-Vertrag.

**Entscheidung (bindend).**
- **Boundaries & Layout.**
  ```
  src/deploy/
    DeployOrchestrator.js   # atomare Einheit Container+Route; Deploy/Undeploy/list
    VpsDockerControl.js      # einziger schreibender Docker-on-VPS-Zugriff (docker pull/run/rm via SSH, ADR-008)
    ReconciliationJob.js     # Cron (ADR-013)
  ```
  `DeployOrchestrator` ist der einzige Ort, der Container- **und** Route-Schritt zu einer Einheit koppelt; er ruft `VpsDockerControl` (Container) und `CloudflareApi` (Route/DNS, ADR-010). `VpsDockerControl` ist der einzige Ort, der **schreibende** `docker`-Kommandos auf einem **VPS** ausführt (read-only-Docker bleibt lokal beim `DockerReader`).
- **Atomare Einheit (Deploy).** `POST /api/deployments {image, vps, hostname}`: (1) `LockoutGuard`-Check auf `hostname` (protected → 422, kein Schritt ausgeführt); (2) `VpsDockerControl` pullt+startet den Container mit dem **Bindungs-Label** (s.u.); (3) `CloudflareApi` legt Tunnel-Route (`PUT …/configurations`) + DNS-CNAME `<hostname>` an. **Rollback-Semantik:** schlägt (3) fehl, wird (2) rückabgewickelt (`docker rm`), so dass **kein verwaister Container** ohne Route zurückbleibt; schlägt (2) fehl, wird (3) nicht versucht. Ergebnis ist `{ result: "ok"|"error", reason? }` ohne Teil-/Geheim-Leak. *(„atomar" = best-effort-Saga mit Kompensation, keine echte 2PC — vertretbar, da der Reconciliation-Cron (C) jeden Rest-Drift einfängt.)*
- **Inverse Einheit (Undeploy).** `DELETE /api/deployments/{vps}/{hostname}` (bzw. via Container-Id): `LockoutGuard`-Check (protected → 422), dann **Route+DNS entfernen** (CloudflareApi, type-to-confirm wie ADR-011), **dann** Container `rm` (VpsDockerControl). Reihenfolge Route-zuerst, damit kein Traffic auf einen bereits entfernten Container zeigt.
- **Container↔Route-Bindung = Label-Konvention (kein State-Store).** Der Container trägt das Label **`cloudflare.tunnel-hostname=<hostname>`** (z.B. `cloudflare.tunnel-hostname=app.example.com`). Das ist die **einzige** Bindung zwischen Container und Tunnel-Route — store-los, von **beiden** Seiten live abgleichbar (Docker-Label ↔ Cloudflare-Route), ADR-005-konform; folgt der Betreiber-/Orchestrator-Empfehlung. Zusätzlich wird das bestehende `preview`-Muster respektiert: Deploys dürfen die `agent-flow.preview`/`agent-flow.compose-project`-Labels koexistent tragen (Wiederverwendung des `preview`-Mechanismus), aber **`cloudflare.tunnel-hostname` ist die maßgebliche Reconcile-Bindung**.
- **VpsDockerControl-Mechanik.** `docker pull "${image}:latest"` → `docker run -d --label cloudflare.tunnel-hostname=<hostname> --restart unless-stopped -p <hostport>:<containerport> "${image}:latest"` → (auf dem VPS) der remote-managed Tunnel routet `<hostname>` → `http://localhost:<hostport>` (Route via CloudflareApi gesetzt). `rm` = `docker rm -f`. SSH-Verbindung + Private-Key store-intern über die ADR-008-`VpsProvisioner`-Linie; **keine** Secrets in Argv/Log/Audit. **OFFENE ENTSCHIEDUNG O3:** Host-Port-Wahl (fixe Konvention vs. erste freie ab 8080 wie `preview`) + ob `VpsDockerControl` den `VpsProvisioner` direkt wiederverwendet oder eine Schwester-Komponente derselben SSH-Linie ist — Detail für die B-Spec/`coder`, beeinflusst keine A/C-Verträge.
- **Kein Deploy-State-Store (ADR-005-Linie).** Die Liste der Deployments wird **live** ermittelt: `VpsDockerControl` listet Container mit `cloudflare.tunnel-hostname`-Label je VPS, `CloudflareApi` listet die Routen — der Schnitt ist der „gesunde" Deploy-Bestand. Kein eigener Deploy-Record.

**Schutz (Floor, prüfbar).** Alle `/api/deployments/*` hinter Access; mutierend zusätzlich `CRED_ADMIN_EMAILS`-Rollenschutz + Audit-First (vor jedem Deploy/Undeploy ein `AuditEntry`: Identität, vps, image, hostname, Aktion). `LockoutGuard`-Hard-Block (ADR-011) auf den Ziel-Hostname **vor** jedem Schritt; Undeploy zusätzlich type-to-confirm. SSH-Private-Key + Cloudflare-Token store-intern, nie geleakt.

**Verhältnis zu ADR-005/006/008/009.** ADR-005 gewahrt (kein Deploy-Store, live aus Docker-Label ⊕ Cloudflare-Route). ADR-006 gewahrt (SSH-Mechanik = ADR-008-Linie, keine neue Provider-/Docker-SDK-Abhängigkeit; Cloudflare via `fetch`). **ADR-008/009 unverändert** — `VpsDockerControl` dockt an die SSH-Linie (008) an und ergänzt den von 009 bewusst ausgeklammerten Container-Layer; **kein** Eingriff in deren Verträge.

**Konsequenzen (prüfbar — Review-Kriterien).**
- `VpsDockerControl` ist der **einzige** Modul, der schreibende `docker`-Kommandos auf einem VPS via SSH ausführt; `DeployOrchestrator` ist der einzige, der Container+Route koppelt (Grep). *(architecture/R01)*
- Jeder Deploy-Container trägt das Label `cloudflare.tunnel-hostname=<hostname>` (testbar: `docker inspect` zeigt das Label). *(architecture/R02)*
- Fehlschlag des Route-Schritts rollt den Container zurück (kein verwaister Container ohne Route — testbar im Saga-Test).
- Deploy/Undeploy auf einen protected Hostname → 422, kein Docker-/Cloudflare-Schritt ausgeführt (ADR-011).
- Kein Deploy-State-Store/keine DB (Grep: kein neuer Persistenz-Pfad außerhalb `CredentialStore`).
- Mutationen hinter Access + `CRED_ADMIN_EMAILS` + Audit-First; SSH-Key/Cloudflare-Token nie in Argv/Log/Audit/Response.

### ADR-013 (Vollfassung) · Reconciliation-Cron (`ReconciliationJob`, beidseitig selbst-heilender Mitternachts-Abgleich)

**Status:** ENTSCHIEDEN · 2026-06-08 · **überarbeitet 2026-06-08** (Betreiber-Korrektur: beidseitige Konvergenz, Selbst-Heilung in beide Richtungen, Report nach jedem Lauf, Panel-Statusmeldung) · Entscheider: `architekt` (Schärfung durch Betreiber bindend vorgegeben) · betrifft [[cloudflare-reconciliation]] (Capability C), [[view-cloudflare]] v2 (Panel-Statusmeldung + Report-Anzeige). Baut auf ADR-002 (always-on dev-gui-Prozess), ADR-005 (kein State-Store; AuditStore als einzige append-only Persistenz), ADR-010 (CloudflareApi), ADR-011 (LockoutGuard), ADR-012 (DeployOrchestrator atomarer Anlege-Pfad + VpsDockerControl + Label-Konvention).

**Kontext.** Betreiber-Vision (geschärft): ein internes „Cron" um Mitternacht geht auf die bestehenden VPS, kontrolliert die laufenden Container, schaut nach, ob es zu einem Container einen Tunnel gibt und umgekehrt, und **bringt beide Seiten zur Konvergenz** auf den Desired-State, der durch das Container-Label `cloudflare.tunnel-hostname` definiert ist: eine Route **ohne** managed Container ist verwaist → entfernen; ein **managed** Container **ohne** Route ist eine unerreichbare Last → Route **anlegen**. So heilt der Abgleich **in beide Richtungen** selbst, auch wenn ein Deploy/Undeploy (ADR-012) mal halb fehlschlägt. Jede Aktion und jeder Lauf wird beobachtbar (Panel-Meldung + persistierter Report).

**Betrachtete Optionen.**

*Option-Achse 1 — Scheduler-Ort:*
1. **Node-interner Scheduler im bestehenden always-on dev-gui-Prozess (ADR-002).** *(gewählt)* dev-gui läuft ohnehin always-on (ADR-002, `restart unless-stopped`); ein interner Timer (z.B. `setInterval`/Cron-artige Berechnung der nächsten Mitternacht) braucht **keinen** neuen Dienst und hält ADR-006 („ein Dienst"). Die Boundaries (`VpsDockerControl`, `CloudflareApi`, `DeployOrchestrator`) sind bereits im Prozess.
2. System-`cron`/`systemd timer` auf dem dev-gui-Host. *Verworfen:* zweiter Steuerort außerhalb des Containers, braucht eigenen Auth-/Token-Zugang zu Cloudflare+SSH (dupliziert das CredentialStore-Setup), widerspricht „ein Dienst".
3. Externer Scheduler (Cloudflare Worker Cron Trigger o.ä.). *Verworfen:* zusätzlicher Dienst + Netz-/Auth-Abhängigkeit, überdimensioniert; widerspricht ADR-006.

*Option-Achse 2 — Heilungs-Richtung bei Drift (Betreiber-Korrektur):*
4. **Container-Label ist autoritativer Desired-State; beidseitige Konvergenz: Route ohne managed Container = verwaist (löschen), managed Container ohne Route = Route anlegen.** *(gewählt — Betreiber-Schärfung)* Der **managed Container** (trägt das Deployment-Label) ist die laufende Realität; sein Label drückt die gewünschte Route aus. Eine Route ohne zugehörigen Container leitet ins Leere → entfernen. Ein managed Container ohne Route ist ein erreichbarkeits-loser, aber gewollter Deploy → die fehlende Route wird **angelegt**, statt nur gemeldet. Beide Heilungen konvergieren auf denselben Desired-State (Label).
5. *(früherer Stand, verworfen durch Betreiber-Korrektur)* Container ohne Route nur **melden**, nicht heilen. *Verworfen:* lässt eine Hälfte des Drifts ungeheilt; der Betreiber will einen **beidseitig** selbst-heilenden Abgleich. Die frühere Sorge „destruktive Auto-Aktion" trifft hier nicht zu — die Heilung „Route anlegen" ist **nicht-destruktiv** (sie killt keinen Container, sie stellt die fehlende Route über denselben atomaren, idempotenten Anlege-Pfad wie ein Deploy her).

*Option-Achse 3 — Anlege-Pfad für die Route-Heilung:*
6. **Wiederverwendung des atomaren Route-Anlege-Pfads des `DeployOrchestrator` (ADR-012, Schritt c).** *(gewählt)* Die Heilung „Route anlegen" durchläuft **genau denselben** Pfad wie ein Deploy (LockoutGuard-Check → `CloudflareApi.putTunnelConfig` + `createDnsRecord`), inklusive derselben Guardrails. **Keine** parallele Anlege-Logik im `ReconciliationJob`.
7. Eigene Route-Anlege-Logik im `ReconciliationJob`. *Verworfen:* dupliziert ADR-012, driftet, eröffnet einen zweiten Cloudflare-Mutationspfad neben dem von ADR-010/011 abgesicherten — verstößt gegen die „eine Stelle"-Linie (R01/R02).

**Entscheidung (bindend).**
- **Komponente & Ort.** `ReconciliationJob` (`src/deploy/ReconciliationJob.js`) läuft als node-interner Scheduler im dev-gui-Prozess (ADR-002). Standard-Trigger: **täglich um Mitternacht** (lokale/UTC-Zeitzone explizit, `coder` finalisiert), plus **manueller Trigger** `POST /api/deployments/reconcile` (Access+Rolle+Audit) für Ad-hoc-Läufe und Testbarkeit. **Der Report entsteht bei beiden Auslösern** (Cron und manuell), nicht nur beim Cron.
- **Ablauf (beidseitig konvergent, idempotent, je VPS/Provider degradierend).** Für jeden **konfigurierten** VPS: (1) `VpsDockerControl` liest laufende Container + deren `cloudflare.tunnel-hostname`-Label (Ist-Container-Menge); Container **mit** Deployment-Label sind „managed", Container **ohne** Label sind **kein Deployment** (nicht geheilt, höchstens als `unmanaged` im Report vermerkt). (2) `CloudflareApi` liest die Tunnel-Routen der betroffenen Zone(n) (Ist-Routen-Menge). (3) **Routen ohne zugehörigen managed Container** UND **nicht protected** (LockoutGuard, ADR-011): **entfernen** (Audit-First, **ohne** type-to-confirm — Cron hat keinen Menschen, ADR-011-Ausnahme). (4) **Managed Container ohne zugehörige Route**: **Route anlegen** über den atomaren Anlege-Pfad des `DeployOrchestrator` (ADR-012, Schritt c: LockoutGuard-Check, dann `CloudflareApi.putTunnelConfig` + `createDnsRecord`; protected Hostname wird **nie** angelegt, als `protectedSkipped` vermerkt; Audit-First). Mehrfacher Lauf ohne Drift = no-op (idempotent, kein Cloudflare-Mutations-Call).
- **Heilung = Wiederverwendung, nicht Duplikat.** Die Route-Heilung in (4) ruft **denselben** atomaren Anlege-Pfad wie der `DeployOrchestrator` (ADR-012) — kein paralleler Cloudflare-Mutationscode im `ReconciliationJob`. Der `ReconciliationJob` orchestriert nur die Drift-Erkennung und delegiert Anlegen/Löschen an `DeployOrchestrator`/`CloudflareApi`. So gelten **automatisch** dieselben Guardrails (LockoutGuard-Hard-Block, Audit-First, Fehler-Semantik). Da der Container für eine managed-Container-ohne-Route-Heilung bereits läuft, ist nur der **Route-Teil** (Schritt c) zu wiederholen — der Container-Teil (Schritt b) entfällt; es findet **kein** `docker pull/run` statt.
- **Idempotenz (beide Richtungen).** Anlegen einer bereits existierenden Route = no-op (CloudflareApi/DeployOrchestrator-Pfad ist idempotent); Löschen einer bereits fehlenden Route = no-op. Ein zweiter Lauf ohne zwischenzeitlichen Drift erzeugt **keinen** Mutations-Call und keine Aktions-Meldung (nur einen Report mit leeren Aktions-Listen).
- **Autorität & Desired-State.** Das **Container-Label** `cloudflare.tunnel-hostname` ist der autoritative Desired-State („soll eine Route auf diesen Hostname existieren"). Eine **protected** Route wird **nie** als verwaist gewertet/gelöscht und ein protected Hostname **nie** angelegt (LockoutGuard hart, in beide Heilungs-Richtungen). Ein **unmanaged** Container (ohne Label) ist kein Deployment, also keine Heilung.
- **Degradation pro VPS/Provider.** Ein unerreichbarer VPS (SSH-Fehler) oder ein degradierter/fehlerhafter Cloudflare-Provider/eine Zone **kippt den Gesamtlauf nicht**: der betroffene VPS/die Zone wird übersprungen und im Lauf-Report als gestört (`errors[]`) vermerkt; die übrigen werden normal abgeglichen (analog ADR-009-Degradation). **Sicherheits-Regel bei Mehrdeutigkeit (fail-closed):** kann der Container-Bestand eines VPS **nicht** zuverlässig gelesen werden, werden für diesen VPS **keine** Routen als verwaist gelöscht (lieber eine verwaiste Route behalten als eine lebende Route löschen) **und keine** Routen geheilt (kein verlässlicher Desired-State). Eine **mehrdeutige Bindung** (zwei managed Container desselben VPS beanspruchen denselben Hostname) wird **nicht** geheilt, sondern als `ambiguous` im Report vermerkt.
- **Interne Panel-Statusmeldung (Vertrag).** Jede Reconciliation-**Aktion** (`route-created` | `route-removed` | `protected-skipped` | `error`) erzeugt eine im Cloudflare-Panel ([[view-cloudflare]]) sichtbare **interne Meldung**. Konsistent mit ADR-005 entsteht dafür **kein** neuer State-Store: die Meldungen werden über die bestehende append-only **`AuditStore`**-Mechanik geschrieben (dieselbe, die jede Mutation auditiert) und dem Frontend als **Read-Model `ReconcileNotice`** über **`GET /api/deployments/reconcile/notices`** (Access; die letzten N, live aus dem AuditStore gefiltert) bereitgestellt — **kein** zweiter Persistenz-Pfad. Eine Meldung trägt `{ at, kind, vps, hostname, detail? }` und **keine** Secrets.
- **Lauf-Report nach jedem Lauf (O4 entschieden: JA).** Jeder Lauf (Cron **und** manuell) erzeugt **genau einen** `ReconcileReport`: Zeit, je VPS/Provider `{ vps, provider?, checkedContainers, createdRoutes[], removedRoutes[], protectedSkipped[], reportedUnmanaged[], errors[] }`. **Persistenz über die bestehende append-only `AuditStore`-Mechanik** (eine Report-Zeile pro Lauf — **kein** neuer Store, ADR-005-Linie; der Report ist Beobachtbarkeit, kein Domänen-State). Lese-Endpunkte: `GET /api/deployments/reconcile/last` (letzter Report) und `GET /api/deployments/reconcile/reports?limit=N` (die letzten N Reports, live aus dem AuditStore) für die Deployments- und Cloudflare-View. Der letzte Report überlebt damit auch einen Neustart (im Gegensatz zur früheren in-memory-Variante).

**Schutz (Floor, prüfbar).** Beide Heilungs-Pfade (Löschen **und** Anlegen) stehen unter demselben `LockoutGuard`-Hard-Block (protected → nie löschen, nie anlegen) + Audit-First wie der interaktive Pfad — durch die **Wiederverwendung** des `DeployOrchestrator`/`CloudflareApi`-Pfads automatisch garantiert. Der manuelle Trigger ist Access+`CRED_ADMIN_EMAILS`+Audit-geschützt. Keine Secrets (SSH-Key/Cloudflare-Token) in Log/Audit/Report/Panel-Meldung/WS/Argv.

**Verhältnis zu ADR-005/006/010/011/012.** ADR-005 gewahrt: der Abgleich ist **live** (Docker-Label ⊕ Cloudflare-Route), kein persistenter Reconcile-/Deploy-State; **Report + Panel-Meldungen nutzen die bestehende `AuditStore`-Append-only-Mechanik** (kein neuer Store/keine DB). ADR-006 gewahrt: kein neuer Dienst, kein neuer Scheduler-Daemon — interner Timer im bestehenden Prozess. ADR-010/011/012 gewahrt **durch Wiederverwendung**: die Heilung nutzt den atomaren Anlege-Pfad (ADR-012) und damit `CloudflareApi` (ADR-010) + `LockoutGuard` (ADR-011) — keine parallele Mutationslogik.

**Konsequenzen (prüfbar — Review-Kriterien).**
- Der Cron läuft **im** dev-gui-Prozess (kein zweiter Dienst/Container für den Scheduler — Grep/Deploy-Artefakt). *(architecture/R13-1)*
- Verwaiste, nicht-protected Route ohne zugehörigen managed Container wird entfernt; protected Route **nie** (testbar mit Mock: protected Route bleibt). *(ADR-011)*
- **Managed Container ohne Route** → Route wird **angelegt** (testbar mit Mock: managed Container ohne Route → Anlege-Call über den ADR-012-Pfad); protected Hostname → **nie** angelegt, als `protectedSkipped` vermerkt. *(ADR-011)*
- **Unmanaged** Container (ohne Deployment-Label) wird **nicht** geheilt, höchstens als `reportedUnmanaged` vermerkt (testbar: kein Anlege-Call für label-lose Container).
- Die Route-Heilung ruft **denselben** atomaren Anlege-Pfad wie der `DeployOrchestrator` (ADR-012) — **kein** eigener Cloudflare-Mutationscode im `ReconciliationJob` (Grep: `ReconciliationJob` enthält keinen direkten `CloudflareApi.mutate*`-Aufruf neben dem geteilten Pfad). *(architecture/R13-2)*
- Unerreichbarer VPS/degradierter Provider → Lauf degradiert (übrige VPS/Provider abgeglichen), und für den gestörten VPS werden **weder** Routen gelöscht **noch** geheilt (fail-closed); mehrdeutige Bindung → `ambiguous`, keine Heilung.
- Idempotenz beidseitig: zweiter Lauf ohne Drift = no-op (kein Cloudflare-Mutations-Call, keine Aktions-Meldung).
- **Jeder Lauf (Cron und manuell) erzeugt genau einen `ReconcileReport`**, persistiert als append-only `AuditStore`-Zeile; abrufbar über `/api/deployments/reconcile/last` + `/reports`. **(O4 entschieden: JA — AuditStore, kein neuer Store.)** *(architecture/R13-3)*
- **Jede Aktion erzeugt eine `ReconcileNotice`** (AuditStore-getragen), abrufbar über `/api/deployments/reconcile/notices`; im Cloudflare-Panel sichtbar. *(architecture/R13-4)*
- Manueller Trigger + beide Heilungs-Pfade: Audit-First, kein Token-/Key-Leak in Audit/Report/Panel-Meldung.
